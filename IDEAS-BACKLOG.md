# Ideas Backlog

Items are prioritized using MoSCoW. Each item should be self-contained — include the "what", "why", and acceptance criteria so it can be picked up without conversation context.

**Format:** `- [ ] **Title** [size] — Description. _Acceptance: ..._`

Sizes: `[S]` < 1 hour, `[M]` 1-4 hours, `[L]` 4+ hours

When work starts: move to In Progress with branch name. When done: move to Done with PR link.

## Should Have

- [ ] **Human signal review and quality feedback loop** [L] — Add the ability for a human to review, accept, or reject individual scanner events, building a labeled dataset that can later be used to improve scanner rules. **Motivation:** Some signals are technically correct (all criteria pass) but qualitatively poor — e.g. the NCPL 2026-04-17 liquidity hunt signal fired after the move had already happened. The scanner can't distinguish "timely" from "too late" without human judgment. **Implementation:** (1) **Backend — new model `SignalReview`**: fields `scanner_event_id` (FK), `verdict` (enum: `accepted`, `rejected`, `uncertain`), `rejection_reason` (nullable enum or free-text — e.g. "too*late", "noise", "stale_data", "split_artifact"), `notes` (optional free text), `reviewed_at` (timestamp). Add endpoints: `POST /api/scanner/events/{event_uuid}/review` to submit a verdict, `GET /api/scanner/events/reviews` to list reviewed events with filters (verdict, scanner_type, date range). (2) **Frontend — inline review controls on scanner results**: On the `ScannerResults` component (`frontend/src/components/ScannerResults.tsx`) and/or `RecentEvents` (`frontend/src/components/RecentEvents.tsx`), add per-event action buttons — thumbs up (accept), thumbs down (reject), question mark (uncertain). On reject, show a small popover/dropdown for the reason category + optional notes. Reviewed events should show their verdict badge inline (green check, red X, gray ?) so the user can see at a glance what's been triaged. (3) **Dashboard view — review stats**: A small summary card showing total signals vs. reviewed, acceptance rate by scanner type, and most common rejection reasons. This helps spot systematic issues (e.g. "40% of liquidity_hunt_pre signals are rejected as too_late"). (4) **Future AI integration (out of scope for v1 but design for it)**: The `SignalReview` table becomes a training/analysis dataset. A future step can query all rejected signals with their `indicators`, `criteria_met`, and `rejection_reason` to ask an LLM or rule engine: "What pattern distinguishes rejected signals from accepted ones? Propose new criteria or threshold adjustments." Design the schema to make this query easy — store the event's `scanner_type`, `ticker`, and snapshot of key indicators alongside the review so it's self-contained. \_Acceptance: (1) User can accept/reject/mark-uncertain any scanner event from the results UI. (2) Rejection requires a reason category. (3) Verdicts persist and are visible on revisit. (4) At least a basic count of reviewed vs. unreviewed signals is shown. (5) No changes to scanner logic itself — this is purely the feedback collection layer.*
