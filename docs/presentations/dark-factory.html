02 · The Idea

Hand the whole gauntlet to an agent — keep the judgement calls.

The Dark Factory is a disposable Docker container that runs Claude Code against a single GitHub issue. You give it one line:

# start a feature from a backlog issue
docker compose --profile factory run --rm dark-factory "Fix issue #3"

It clones the repo, branches, does the work, stands up a live preview, opens a PR with the preview URL — then exits. Nothing touches your machine.

①You say what, not how

A GitHub issue is the entire interface. No prompts, no babysitting.

②It works in a sandbox

Fresh clone, non-root, no host access, walled off from the live stack.

③You review the result

A real PR and a running preview you can click through. Humans approve every gate.

⟳The scheduler

Polls the board every 30s and runs a priority waterfall — In Review → Blocked → Backlog → Refined → Ready. One dispatch per cycle, honouring the board's WIP limits.

✋Three human gates

Approve the spec (→ Refined), approve the plan (→ Ready), review the PR (→ Done). You move the card or comment — nothing merges itself.

🏷Labels & guards

needs-discussion pauses automation; epic is decomposed by hand; failures drop to Blocked and retry up to 3×.

06 · Security Model

Capability, not trust.

The factory runs with dangerously-skip-permissions internally — which is only safe because the blast radius is engineered to near-zero.

What it CAN do

  • Read GitHub issues & PR review comments
  • Clone the repo, push branches, open PRs
  • Create / tear down mh-preview-* stacks
  • Call the Anthropic API & package registries

    • What it CANNOT do

  • Read or write files on the host machine
  • See or reach the live MarketHawk stack
  • Touch the Docker daemon directly
  • Run privileged / capability-escalated containers
  • Modify its own read-only credential file

    07 · Orchestration

    A declarative workflow routes the whole thing.

    The container runs one Archon workflow — archon-dark-factory.yaml — a DAG of nodes with conditional routing. The first node reads your command and decides the path.

    →Intent router

    newcontinuecloserefineplan

    One verb selects which branch of the DAG runs. Epics auto-resolve to their next open sub-issue.

    # nodes (excerpt)
    - id: parse-intent  model: haiku
    - id: fetch-issue  gh + epic resolve
    - id: refine    when: intent==refine
    - id: plan      when: intent==plan
    - id: implement when: new | continue
    - id: preview-up  isolated stack
    - id: validate   smoke tests
    - id: push-and-pr draft PR
    model tiering → haiku parses sonnet refines · plans · builds

    Cheap models do mechanical steps; capable models do the reasoning. Every step is logged with token cost.

    09 · Deep Dive — Refine (the Spec)

    An orchestrator brainstorms; a product-owner subagent answers.

    • One question at a time. The orchestrator runs the brainstorming skill and formulates a single clarifying question.
    • A subagent answers it. For each question it spawns a product-owner subagent with Glob/Grep/Read to explore the codebase and decide.
    • It can say "I don't know." A subagent may return UNCERTAIN: → the factory posts the question and adds needs-discussion, pausing for a human.
    • Then it writes the spec. Proposes 2–3 approaches, picks one, writes Docs/superpowers/specs/…, self-reviews, commits.
    • Human gate. Posts the full Q&A to the issue and labels it spec-pending-review.
    REAL OUTPUT · ISSUE #143
    Q: Use the Archon loader, or a lightweight YAML parse?
    A: The incident was a raw syntax error — yaml.safe_load catches the actual failure class with zero new tooling in CI.
    Q: New CI job, or a step in an existing job?
    A: A step in the test job — the check is sub-second; a new job adds ~30s of runner overhead.
    Q: Validate all files, or only changed ones?
    A: All files — a bad rebase can break a file outside the diff.
    3 questions asked → spec generated → spec-pending-review
    10 · Deep Dive — Plan (Architect-validated)

    Every plan is reviewed by an architect before a human ever sees it.

    • Bite-sized & test-first. writing-plans emits tasks where each step is: write a failing test → confirm it fails → implement → confirm it passes → commit.
    • No placeholders. Exact file paths, real code blocks, expected command output.
    • Architect review loop. An architect subagent validates the plan against the spec — up to 3 cycles until Approved.
    • Stuck → human. Not approved after 3 cycles? Posts the feedback and labels needs-discussion.
    • Human gate. Labels plan-pending-review.
    spec→ draft plan→ architect: issues?↺ approved

    11 · Deep Dive — Implement (Multi-agent Build)

    The build is a red-green-refactor loop, not a code dump.

    1. write failing test→ 2. confirm it fails→ 3. minimal code→ 4. confirm it passes→ 5. small commit↺

    ⤺Feedback-driven continue

    On Continue issue #N it reads your PR review comments — including line-level inline notes — and changes only what you asked for. It never re-implements from scratch.

    🗄Migrations & seed data

    New model → autogenerate + apply an Alembic migration. Feature needs data the preview lacks → it writes idempotent seed SQL.

    📚Docs stay honest

    It maps changed files to ARCHITECTURE.md / PROJECT_STRUCTURE.md sections and updates them in the same PR.

    Then it stands up the preview and curl-tests the changed endpoints live before declaring done — verification before completion, enforced.

    14 · The Self-Healing Story

    #142

    root cause found & the workflow repaired
    ›

    Prevented
    factory builds the CI gate that blocks unparseable YAML

    🔍Why CI missed it

    The pipeline never parsed Archon's workflow definitions, so a syntax error sailed through review and broke discovery for a day.

    ♻The system closes the loop

    The fix — a YAML-parse gate in CI — is itself being specced, planned and built by the factory (#143). The tool hardens the tool.

    19 · Where This Goes

    From a tool — to a platform.

    Nothing here is MarketHawk-specific. The same container, pointed at any repo, turns a backlog into shipped, reviewed code.

    🌐Any repo, any team

    The factory reads CLAUDE.md and the board. Point it somewhere new and it learns that codebase's conventions.

    📐The constraint moves

    The bottleneck stops being engineering-hours. It becomes idea quality and review bandwidth — the things only humans supply.

    🧭Humans curate

    People decide what is worth building and whether it's right. The factory handles everything in between.

    We are already on the path — #143 hardening, #104 registry & CD, #141 Kubernetes for a cloud go-live.
