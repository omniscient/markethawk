01 · Two brains, one bus

A conductor and an assembly line that only talk through the board.

  • Scheduler — decides what & when. Polls the board, runs the priority waterfall, enforces WIP, retries, and the CI/conflict guards. It never writes code.
  • Archon — does how. Inside each ephemeral container it parses intent, branches, and runs refine / plan / build / conformance. It never decides scheduling.
  • The GitHub board + labels is the only thing they share — and it's also the human's control surface.

⟳WIP limits live on the board

The scheduler reads limit: N from each Status column's description at startup — In Progress and In Review caps are data, not code.

🔗Dependencies in the body

A Depends on: #NN line blocks a Ready item until that dependency reaches Done.

🏷Labels are the API

Both brains and the human coordinate entirely through column + label transitions — no side channel, no shared database.

04 · The waterfall, explained

Why this order? Finish work before starting work.

P0
CI-failure gate · always
Red-CI PRs in review → Blocked + comment, so a human can't merge a failing PR. Runs even when a container is busy.
⌁
Orphan sweep · always
In-Progress with no live container (host reboot / OOM) → Blocked for retry.
P1.5
Conflict scan → Deconflict
In-Review PR is CONFLICTING → resolve before a human looks at it.
P1
Review comments
New human comment → classify → Merge / Continue / Skip. Unblock work already in flight.
P2
Ready → Fix
Implement what's already refined + planned (respects WIP + dependencies).
P3
Blocked → retry
Branch-aware: Continue if a PR exists, else Fix. Circuit-break after 3.
P4
Refined → Plan
Advance refined work before pulling brand-new backlog.
P5
Backlog → Refine
Lowest priority: groom new (opt-in) ideas only when nothing else needs attention.

One issue, its whole journey across columns and gates.

✋Three gates

Spec → Refined, Plan → Ready, PR → Done. A human moves the card (or comments to iterate).

⏩direct-to-pr auto-advance

With that label the spec & plan gates self-open after a 30-min grace window, and the end-gate auto-Closes on PR approval / auto-Continues on changes requested.

⏸needs-discussion

The universal off-switch — an issue carrying it is invisible to every dispatch loop until a human removes it.

06 · Resilience — how stuck work recovers

Nothing sits stuck silently. Three escape hatches.

failure → recoveryflow
flowchart TD
  F["run fails / dies"] --> T{"trappable exit?"}
  T -->|"yes"| OF["entrypoint on_failure → Blocked + comment"]
  T -->|"no — OOM / reboot"| SW["scheduler orphan sweep → Blocked"]
  OF --> RETRY{"retry count < 3?"}
  SW --> RETRY
  RETRY -->|"yes"| RE["re-dispatch: Continue if PR else Fix"]
  RETRY -->|"no"| CB["circuit-breaker: trip_to_blocked"]
  CB --> ND["+ needs-discussion · reset counter · post comment"]
  ND --> HUMAN(["👤 human investigates"])
  RE --> F

🔁Per-phase retries

Counters are keyed by phase — #N:refine, #N:plan, #N:resolve, #N (implement) — so a flaky plan doesn't burn the build budget.

⛔Circuit-breaker

After 3 failed attempts in a phase: Blocked + needs-discussion + an explanatory comment with the manual retry command, then the counter resets clean.

🧹Durable state

Retry counts live on a named volume, so they survive the restart: unless-stopped cycle of the scheduler container.

📐Bite-sized & test-first

Every task is: failing test → confirm fail → implement → confirm pass → commit. Exact paths, real code, expected output — no placeholders.

🏛Architect loop

An architect subagent validates against the spec, up to 3 cycles. Still not approved? Post the feedback and needs-discussion.

✔Plan-vs-spec gate

A second reviewer checks the plan conforms to the spec (Gate 1) — material divergence reconciles or blocks.

⛔Scope discipline

Spot a pre-existing bug while building? Do not fix it. Record it in out-of-scope.md — the conformance gate turns each entry into a backlog ticket.

⤺Continue = feedback-driven

On Continue #N it reads PR review + inline comments (with path/line) and changes only what was asked.

🗄Migrations · seed · docs

New model → autogenerate Alembic migration + __init__ import. Missing preview data → idempotent seed SQL. Docs map kept honest in the same PR.

12 · Conformance & scope gate (Gate 2 — code ↔ spec)

Does the code match the spec — and only the spec?

dark-factory-conformanceactivity
flowchart TD
  D["pre-triage diff: strip lockfiles / docs / memory"] --> REV["reviewer: code vs spec"]
  REV --> OOS{"out-of-scope changes?"}
  OOS -->|"yes & enforce"| EX["excise from branch + file scope-spillover ticket"]

  OOS -->|"no"| V{"verdict"}

  RC -->|"yes"| FIX["failing test → fix → commit"]
  FIX --> REV
  RC -->|"no"| BLK["Blocked + needs-discussion"]

🔍Pre-triage first

Lockfiles, generated indexes and memory are stripped from the diff so they aren't mis-flagged as out-of-scope.

✂Scope excision

Out-of-scope hunks are reverted from the branch and each becomes a linked scope-spillover ticket — the branch stays clean, the defect isn't lost.

⛔Block on material

A material divergence reconciles in ≤3 TDD cycles; failing that, the issue goes Blocked rather than shipping a PR that doesn't match its spec.

🧭Classifier, not keywords

A haiku pass reads comments since the last run and returns MERGE/CONTINUE/SKIP. When in doubt between Continue and Skip, it chooses Continue.

🤖Bot comments ignored

The factory's own cost reports and pipeline status comments are matched by footer marker and never re-trigger work (the fix for the #124 spec loop).

⏩Hands-off mode

For direct-to-pr issues the end-gate reads the PR's GitHub review state directly: APPROVED → Close, CHANGES_REQUESTED → Continue.

15 · Preview environments — only when they earn it

A clickable stack per issue — if the change touches the app.

  • Differentiator first. A haiku classifier reads the diff: docs / config / tests / CI-only → skip preview. Anything under backend/app, frontend/src, migrations, compose → build it. Fail-safe builds on any doubt.
  • Collision-free slots. Scans running stacks and picks the lowest free slot 00–99 — decoupled from the issue number (#124).
  • Self-bootstrapping. Fresh DB → create tables, stamp + upgrade. Existing DB → upgrade only. Health-gated on /api/health.
  • Reachable by hostname. The factory joins the preview network and curl-tests endpoints by container hostname, then tears down on Close.

♻Ephemeral by default

Every run is a throwaway --rm container working from a fresh clone — nothing persists between runs, so nothing accumulates to compromise.

📥Your source is never mounted

The agent edits a clone at /workspace/gitrepo, not your host working tree — it cannot read or write your real files.

🧪Previews isolated & synthetic

Each mh-preview-N is its own network + a DB seeded with synthetic data, torn down on Close — no production data, no cross-talk.

🎯Bounded blast radius

One run at a time (hard lock), runtime-only credentials never baked into the image, and a narrow capability surface: git, GitHub, Anthropic, preview orchestration.

20 · One issue, every failure mode

However it breaks, there's a path back to motion.

CIPR goes red after merge to main elsewhere

P0 pulls it out of review → Blocked; P3 continues the existing branch and re-runs validate to fix it.

⚔Branch drifts into conflict

P1.5 spots CONFLICTING and dispatches Deconflict — tiered resolution pushes a clean branch back to review.

💀Run is SIGKILLed mid-build

No error handler fires, but the orphan sweep finds the In-Progress card with no container and routes it to Blocked for retry.

3×It just can't get there

After three attempts in a phase the circuit-breaker stops the bleeding: Blocked + needs-discussion + a comment telling a human exactly how to resume.

The common thread: failures converge on Blocked, and Blocked is just another column the scheduler knows how to work.

21 · Where it goes next — industrialization

From embedded tool to a multi-repo platform.

Today the factory lives inside one git repo. The path to a product is to lift it out and make every assumption configurable.

⧉Extract & generalize

Lift the factory + scheduler out of this repo into a standalone, repo-agnostic control plane — one deployment driving many repositories, each with its own config.

☁Kubernetes & cloud

Move runs and previews off a single host's Docker engine onto Kubernetes — the factory scales horizontally and runs as a managed cloud service.

🔌Pluggable trackers

Abstract the board + labels bus behind an adapter so it isn't GitHub-only — Jira (and GitLab, Linear…) as first-class backends.

Issue in. Reviewed PR out. Next — for every repo, on any tracker.
