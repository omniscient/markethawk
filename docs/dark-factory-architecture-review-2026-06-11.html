What frontier practice says (2025–26)

DORA 2025Added Rework Rate as a fifth metric precisely because AI inflates first-pass throughput and pushes correction downstream. Small batch size is the strongest lever. dora.dev
Cognition '26PR merge rate is the industry top-line (Devin: 34%→67% YoY — moved by harness/scoping iteration, not model upgrades). cognition.ai
METR '26Maintainers accepted only ~half of AI PRs that passed automated graders — test-passage overstates mergeability ~2×. Track success by ticket-size bucket to find your autonomy ceiling. metr.org
GitClear '25The slop proxies: 2-week churn on AI-authored lines, copy/paste vs moved-line ratio, duplicate-block growth. All computable from git history alone. gitclear.com
MSR '26Post-merge SonarQube on 1,210 agent PRs: PR size drove issue density, not which agent. Recommends size-aware static-analysis gates beyond unit tests. arXiv 2601.20109
pass^kFor unattended pipelines, pass^k = (c/n)^k is the honest reliability metric: 70% single-run success → only 34% chance of 3 clean runs in a row. philschmid.de
Judge calibrationLLM judges usable as gates at ≥75% human-agreement; production judges missed ~80% of guardrail failures in one 2026 study. Deterministic checks hard-gate; judges advisory until calibrated. arXiv 2606.10315
Anthropic '26Build the first eval set from 20–50 of your own production failures; long-running harnesses must smoke-test the baseline before starting new work. anthropic.com
ACE / ICLR '26Memory as an append-only playbook of small delta lessons; never let an agent wholesale-rewrite its own lessons file (context collapse). Validates the .archon/memory design, prescribes its governance. arXiv 2510.04618

C1 · The Run Record — give every factory run one deep artifact

Strong in-process

dark-factory/entrypoint.sh · dark-factory/scheduler.sh · .archon/workflows/archon-dark-factory.yaml · Seq (ADR-0011)

Before — run state scattered across 5 sinks, 4 of them lossy

(current state, no history)"]
  R --> E["Seq raw lines
(unstructured GELF)"]

After — one run-record module; every stage writes a verdict through one interface

flowchart TD
  S1[implement] --> RR
  S2[validate] --> RR
  S3[conformance] --> RR
  S4[code-review] --> RR
  S5[scheduler] --> RR
  RR["RunRecord
record(stage, verdict, tokens, ms)"]:::deep
  RR --> Q["Seq (structured,
gen_ai.* attributes)"]
  RR --> J["runs.jsonl

  classDef deep fill:#0f172a,color:#fff,stroke:#0f172a;

Problem: the factory produces verdicts everywhere and records them nowhere — conformance status, review findings, validation results, retries, and durations are split across issue comments, ephemeral /tmp files, a retry-counter JSON, and unstructured Seq lines; this whole review's baseline had to be reconstructed by hand.

Solution: a run-record module with one interface — record(stage, verdict) — that every stage and the scheduler write to; persisted as one structured record per run (issue, intent, stages, outcomes, tokens, cost, durations) shipped to Seq with OTel gen_ai.* attribute names and appended to a queryable history file.

  • leverage: one interface feeds every metric in C2, C3, C9
  • locality: gate-effectiveness questions become one query
  • verdict artifacts stop dying with /tmp
  • future-proof: OTel GenAI schema, not a private one

Problem: every change to the DAG, a command prompt, or a gate threshold ships unmeasured — the OR-join trigger_rule self-regression (ce9e4a3) proved the harness can silently break itself, and there is no fixed suite to catch the next one.

Solution: a replay suite of 10–20 already-closed issues (benchmark-eligible iff their merged PR contains tests that fail pre-patch and pass post-patch), each run 3× against the pre-PR commit; report pass^k per size bucket; re-run the suite whenever the factory's own prompts or DAG change.

  • the interface is the test surface — for the factory itself
  • pass^k is the honest unattended-reliability number
  • harness regressions caught before they burn live runs
  • Max-plan token cost is the real constraint: run nightly or pre-change, not per-commit

C4 · Deepen the Gate seam — layered policy, calibrated judges

Strong local-substitutable

.archon/commands/dark-factory-conformance.md · dark-factory-code-review.md · .claude/skills/refinement/config.yaml · codeindex get_impact

Before — two shallow gates, miscalibrated in opposite directions

conformance — over-blocks: 9 excise-reverts, 1 double-revert, 42 spillover tickets
inline route_memory_file() + write_memory_entry()
code-review — under-blocks: 0 blocking findings across all sampled PRs; fail-open; auto-advance
inline route_memory_file() + write_memory_entry() (95% duplicate)

PR #278 passed both gates and re-reverted a settled migration decision.

After — one Gate interface, a policy cross-section

hard gate · deterministic — tsc, pytest, migrations, conflict-marker grep
hard gate · structural — PR-size budget + blast-radius (get_impact on hotspots ⇒ human approval)
advisory · LLM judges (conformance, code-review) — promoted to blocking only at ≥75% agreement on 20–30 hand-labelled past verdicts

Both judges become adapters of one Gate interface: verdict(diff, context) → {status, findings[], severity} + one shared memory-write implementation. Two adapters = the seam is real.

Problem: the two LLM gates are shallow modules — near-identical inline memory-write implementations, separate verdict formats, no shared policy — and neither is calibrated: conformance generates churn (the reverts), code review rubber-stamps (zero blocks), and the gating order doesn't distinguish deterministic facts from model opinions.

Solution: one Gate interface with conformance and code-review as adapters; deterministic and structural checks hard-gate, LLM verdicts stay advisory until a one-time calibration audit (hand-label 20–30 past verdicts) earns them blocking power; high-blast files (codeindex get_impact) always require human approval regardless of judge verdict.

  • locality: one memory-write implementation, one verdict format
  • leverage: new gates (security, perf) plug into the same seam
  • #278-class incidents need a deterministic or human gate, and get one
  • calibration turns "0 blocking" from suspicious to verified
Overlaps open work: #276 (scope over-reporting) and #275 (root cause) cover the conformance half; the Gate interface + calibration audit is the part nothing currently tracks.

C5 · Baseline-green gate — never start work on a red main

Strong cheap

.archon/workflows/archon-dark-factory.yaml (new stage 0) · dark-factory/scheduler.sh

Before — issue #250, June 5–6

t+0h
PR #228 merges a tsc break to main
t+1h
every dispatched ticket fails at preview-up
t+3h
circuit breakers trip across the backlog, runs burned
t+48h
human diagnoses via Seq + fresh tsc, fixes main

After — stage 0 smoke gate

t+0m
smoke main: tsc -p tsconfig.app.json + backend import check (~2 min)
red?
halt run · file/flag one "main is red" ticket · scheduler pauses dispatch
green
proceed; any later tsc failure is provably the factory's own

Problem: the pipeline assumes main is green; when it isn't, every run fails at preview-up for someone else's reason, the circuit breaker mis-attributes the failure to each ticket, and runs burn until a human notices — 48 hours, in the one measured incident.

Solution: a stage-0 smoke check of origin/main (the exact commands the memory file already prescribes for diagnosis) that halts the run, files one root-cause ticket, and pauses the scheduler when red — Anthropic's published harness pattern ("verify the baseline before new work"), learned here the hard way and not yet codified.

  • failure attribution becomes trustworthy (own break vs inherited)
  • circuit-breaker state stops lying
  • ~2 minutes per run buys 48-hour-class incident immunity

Problem: board updates, retry/circuit-breaker logic, and the three-tier de-conflict live as duplicated bash across entrypoint, scheduler, and inline DAG yaml — the deletion test says deleting any one file scatters complexity to the others, and none of it is testable, so factory bugs (the trigger_rule regression, the double-revert) are only caught in production runs.

Solution: deepen into one Python factory-core module (board ops, retry state, de-conflict tiers, run-record from C1) with a pytest suite; entrypoint.sh, scheduler.sh, and DAG stages become thin adapters calling it — the interface stays the same shell commands, the implementation concentrates.

  • locality: one place for retry/board/conflict bugs
  • the factory's own changes become testable pre-merge (pairs with C3)
  • two adapters (entrypoint, scheduler) prove the seam today
Overlaps #215 (scheduler tests) — but #215 tests the bash as-is; this concentrates first so the tests have one interface to hit. Note: baked-image constraint applies — core changes need docker compose build (memory: scheduler/entrypoint are baked).

C7 · One config interface

Worth exploring in-process

.claude/skills/refinement/config.yaml · scheduler.sh env defaults · entrypoint.sh env defaults · SEVERITY_ORDER in code_review_payload.py

Before — same knob, three owners

config.yaml: refine.wip_limit: 2
scheduler.sh: REFINE_WIP_LIMIT=2 (independent default)
code_review_payload.py: SEVERITY_ORDER hardcoded (diverges from block_threshold enum)

After — yaml is the interface, everything reads it

config.yaml — single source; scheduler/entrypoint read via yq at startup; env vars demoted to overrides; payload script reads the same enum

Interface = the documented knob list. Implementation = wherever each consumer reads it. Today the interface is as wide as three implementations.

Problem: the config surface is shallow — the same knobs are defined independently in yaml, two shell scripts' env defaults, and a Python constant, so they silently diverge and there is no single place to answer "what is the factory's current policy?"

Solution: make config.yaml the one interface — scheduler and entrypoint parse it at startup (env vars become explicit overrides, logged when active), and the severity enum lives in the yaml that block_threshold already references.

  • deletion test: deleting yaml today loses only some knobs — after, it loses all (that's depth)
  • policy questions get one answer

Problem: when the circuit breaker trips, the failure's cause evaporates — the human reconstructs it from Seq, the retry run starts blind, and nothing accumulates; the factory's most valuable data (its own failures) is discarded.

Solution: on circuit-break, a cheap haiku post-mortem step writes a one-paragraph "why this failed" comment on the ticket; the retry reads it (the published Reflexion pattern), and each factory-caused failure is promoted into the C3 eval suite — Anthropic's "build evals from your real failures" loop, automated.

  • retries stop repeating the same mistake
  • the eval suite grows from reality, not invention
  • human Seq forensics becomes the exception
Complements #254 (memory governance): post-mortems are per-ticket and ephemeral by default; only twice-confirmed lessons graduate to .archon/memory — exactly the ACE "delta items, never wholesale rewrite" discipline.

C9 · Size- and type-aware dispatch policy

Speculative needs C2 data first

scheduler.sh dispatch loop · existing size: S/M/L and priority labels

Hypothetical success-by-size (illustrative — C2 will give real bars)

S
~high
M
~mixed
L
~burns runs

MSR '26: chores merge at 84%, perf work at 55% — task type matters as much as size.

Policy

below ceiling (S, chores, tests, docs) → full autonomy, direct-to-pr
at ceiling (M, features) → autonomy with plan-pending-review gate
above ceiling (L, perf, architectural, migrations) → refine-only; human pairs on implement

Problem: the scheduler dispatches every ready ticket identically, but agent success rates are known to vary ~30 points by task type and collapse above a size threshold — undisciplined L-ticket dispatch is where 5-hour Max windows go to die.

Solution: once C2 produces success-by-bucket numbers, encode a dispatch ceiling: full autonomy below it, gated autonomy at it, refine-only above it — METR's time-horizon idea applied as an internal dispatch policy.

  • runs concentrate where the factory actually succeeds
  • the ceiling is measured, revisited quarterly as models improve

Calibration verdicts: artifacts & reviews

Artifacts — roughly right amount, wrong persistence

  • Keep: spec, plan, cost report, run report comment, spillover tickets.
  • Fix: the four verdict artifacts (validation, conformance, review, conflict-resolution .md) die in /tmp — attach them to the PR or fold into the C1 run record.
  • Trim: spillover ticket auto-filing is over-producing (42 tickets, many duplicate/cosmetic — already tracked as #275/#276). Dedup-before-file beats file-then-triage.

Reviews — right count (2 gates + human), wrong calibration

  • Don't add more review stages — fix the two you have (C4): conformance over-fires, code-review never fires.
  • Human review is currently optional in practice (auto-advance + grace windows). Make it mandatory only where it pays: blast-radius hotspots, migrations, seed data, auth.
  • The 30-min grace window auto-advance is fine for S tickets; for the high-blast set it's how #278 happened.

Top recommendation

Start with C1 + C2: the run record and the Factory Scorecard

Every other candidate — judge calibration (C4), the dispatch ceiling (C9), the replay benchmark (C3), the failure flywheel (C8) — consumes measurements that don't exist yet. C2 is a day of labels and scripts and immediately answers "is the factory good?" with industry-comparable numbers; C1 is the deep module that makes those numbers a query instead of an excavation. You cannot calibrate what you don't record — and the single most expensive current behavior (gates miscalibrated in both directions) is invisible without them.

Sequence: C2 (days) → C5 (one stage, cheap insurance) → C1 (the seam everything else plugs into) → C4 → C3 → C8 → C9. C6 and C7 ride along whenever the glue is next touched.
