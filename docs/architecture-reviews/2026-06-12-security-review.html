Overall Health

MarketHawk is materially more hardened than a typical internal trading dashboard, and notably more than its own earlier state: it now has real authentication. A global pure-ASGI AuthMiddleware validates a JWT (HS256) carried in an HttpOnly; Secure; SameSite=strict cookie on every request except a small static exempt list; a CSRFMiddleware enforces a CSRF check on state-changing routes; slowapi rate-limits sensitive endpoints; config fails fast when JWT_SECRET_KEY is shorter than 32 chars; stack traces are hidden unless ENVIRONMENT=development; and database access is uniformly through the SQLAlchemy ORM with bound parameters (no string-built SQL in application code).

Residual risk concentrates in four areas: (1) this is not read-only software — it can place live bracket orders at Interactive Brokers (ib.placeOrder), gated only by configuration defaults with no hard notional/qty ceiling or order-layer kill switch; (2) it is single-user with no authorization model (no user_id on any table); (3) several surfaces are exposed-by-default (Swagger/openapi.json//metrics auth-exempt, Redis without a password, admin tools relying on un-enforced env credentials); and (4) supply-chain gaps in the build/factory path (curl | bash installers, unpinned images/actions, advisory-only Trivy). No hardcoded production secret was found in the tree; no SQLi, command injection, unsafe deserialization, or template injection in application code; no dangerouslySetInnerHTML in the frontend.

Bottom line: safe to run for a single trusted operator on a trusted host/LAN today. Before any internet-facing or multi-user deployment, close the Top-10 list — chiefly the admin-tool / /metrics / docs exposure, Redis auth, and explicit trade-execution guardrails.

Confirmed defensive controls (working)

  • Fail-fast secret validation: config.py:137-145 rejects JWT_SECRET_KEY < 32 chars; DATABASE_URL scheme validated.
  • Global deny-by-default auth: pure-ASGI AuthMiddleware (main.py:282-312) + CSRFMiddleware + SameSite=strict, COOKIE_SECURE=True default.
  • ORM-only data access; bound parameters throughout. Only f-string SQL is in a seed migration with hardcoded data.
  • No pickle/yaml.load/eval/exec; Celery uses the JSON serializer.
  • Circuit breakers on Polygon and IBKR (pybreaker) bound retry storms.
  • Tokens in HttpOnly cookies (not localStorage); all 3 WebSocket routes authenticate via ws_get_current_user.

5. Detailed Findings

F-TRADE-01 — Live order placement with config-only guardrails

HIGH · Confidence High · OWASP A04:2021 Insecure Design · CWE-840

Affected: providers/ibkr_orders.py:205-299 (place_bracket_order → ib.placeOrder @276); services/auto_trade_service.py:123-279; tasks/trading.py; routers/auto_trading.py.

Why it matters: the highest-impact capability in the codebase — it can move real money. The review brief asked us to confirm read-only vs execution: confirmed, it executes OCA-linked bracket orders (entry + stop + target).

Controls present (good): strategies default paper_mode=True; a SystemConfig.AUTO_TRADING_ENABLED flag must be on; orders can require manual approval. Paper orders simulate fills and never reach placeOrder.

Gap / abuse case: guardrails are all configuration and single-user. No hard order-layer ceiling on quantity/notional, no daily-loss kill switch inside place_bracket_order. Anyone who authenticates (single shared credential) can flip the three flags and the system trades the live account unattended off scanner signals.

Remediation: add a non-bypassable guard at the top of place_bracket_order (notional/qty cap + TRADING_KILL_SWITCH); make max_position_usd required when paper_mode=False; require a separate LIVE_TRADING_ARMED flag not settable from the API; WARN-log + Prometheus-count every live placement.

if os.getenv("TRADING_KILL_SWITCH","").lower() in ("1","true","yes"):
    raise PermissionError("Trading kill switch engaged")
notional = quantity * (entry_price or target_price)
if notional > settings.MAX_ORDER_NOTIONAL or quantity > settings.MAX_ORDER_QTY:
    raise ValueError(f"Order exceeds hard cap: notional={notional} qty={quantity}")

F-AUTH-01 — Swagger / ReDoc / openapi.json + /metrics unauthenticated in production

HIGH · Confidence High · OWASP A05:2021 Misconfiguration · CWE-200

Affected: main.py:266-275 — EXEMPT_PREFIXES includes /docs, /redoc, /openapi.json, /metrics.

Why it matters: the global AuthMiddleware is otherwise deny-by-default, but these prefixes are whitelisted — so the entire API schema (every route, param, model) and Prometheus operational metrics are world-readable wherever the backend is reachable. openapi.json is a turnkey recon map; /metrics leaks request volumes, WS connection counts, internal timing — all pre-auth.

Remediation: disable docs in prod (docs_url=None unless ENVIRONMENT=development); require auth (or Caddy basic-auth / IP-allowlist) on /metrics; remove these four from EXEMPT_PREFIXES in prod.

docs_url = "/docs" if settings.ENVIRONMENT == "development" else None
app = FastAPI(..., docs_url=docs_url, redoc_url=docs_url,
              openapi_url=("/openapi.json" if docs_url else None))

F-NET-01 — Redis has no authentication

HIGH · Confidence High · CWE-306 · CIS Docker network controls

Affected: config.py:24 (REDIS_URL = "redis://redis:6379/0" — no password); Redis service in docker-compose.yml has no requirepass. Used as Celery broker and result backend and cache and pub/sub.

Abuse case: any process on the docker network can read cached data, inject/forge Celery tasks (e.g. enqueue execute_auto_trade with attacker rule/event ids), and publish to live WS pub/sub channels (tweet_signals:all, watchlist, news) that fan straight out to authenticated browser clients.

Remediation: set requirepass (ideally ACLs); change REDIS_URL to redis://:${REDIS_PASSWORD}@redis:6379/0 everywhere; never publish 6379 beyond 127.0.0.1. Keep task_serializer='json' explicit.

F-ADMIN-01 — Admin tools depend on un-enforced env credentials

HIGH · Confidence Medium · OWASP A05:2021 · CWE-1188 / CWE-798

Affected: docker-compose.yml (pgAdmin :5050, Flower :5555, Seq :5380, Grafana :3001); .env.example (change_me_pgadmin_password, FLOWER_BASIC_AUTH, SEQ_ADMIN_PASSWORD_HASH).

Why it matters: these expose database contents (pgAdmin), task payloads/logs (Flower), centralized logs incl. stack traces/SQL (Seq), and dashboards (Grafana). If FLOWER_BASIC_AUTH or SEQ_ADMIN_PASSWORD_HASH is left empty, those tools start with no auth — and there is no fail-fast. (The recreate in this very session emitted FLOWER_BASIC_AUTH variable is not set.)

Remediation: refuse to boot each admin service when its credential env is empty or matches a change_me_* placeholder; bind all admin UIs to 127.0.0.1 and reach them via SSH tunnel / Caddy basic-auth + IP allowlist; never expose to 0.0.0.0.

F-AUTHZ-01 — No authorization model; single-user; no user_id on data tables

MEDIUM · Confidence High · OWASP A01:2021 Broken Access Control · CWE-639/862

Affected: core/auth.py (binary authentication only); all models/* (no user_id FK on Trade, journal, ScannerConfig, StockUniverse, AutoTradeOrder, AlertRule).

Why it matters: any valid token can read/modify every record, trigger every task, and arm trading. The schema cannot express ownership, so adding a second user silently shares all journals, watchlists, and strategies (horizontal priv-esc by construction).

Remediation: before multi-user, add user_id FKs, scope every query by current_user.id, and add a role concept (≥ admin vs viewer) gating trading/system-config endpoints. Until then, document the single-user constraint and keep the deployment single-tenant.

F-SUPPLY-01 — Floating image tags + unpinned base images / actions

MEDIUM · Confidence High · CIS Docker 4.x · GitHub Actions hardening / SLSA

Affected: docker-compose.yml:330,352,371,426 (pgadmin4:latest, seq:latest, seq-input-gelf:latest, docker-socket-proxy:latest); Dockerfiles (FROM python:3.12-slim, node:22-alpine, ubuntu:24.04 — tags, no digests); .github/workflows/* (actions pinned to major version, not SHA).

Why it matters: non-reproducible builds; a retagged or compromised upstream image/action silently enters the next rebuild. The factory rebuilds images, so drift is live.

Remediation: pin compose images to explicit versions; pin Dockerfile FROM to @sha256: digests; pin Actions to commit SHAs; add Dependabot/Renovate.

F-SUPPLY-02 — curl | bash installers + unsigned fork clone in factory image

MEDIUM · Confidence High · CWE-494 · supply-chain best practice

Affected: dark-factory/Dockerfile:22 (curl … nodesource … | bash -), :41 (curl … bun.sh/install | bash), :69 (git clone -b feat/… omniscient/Archon.git).

Why it matters: remote code executed at build time with no signature/hash verification; a MITM or upstream compromise injects code into the factory image, which has docker-socket-proxy access (BUILD/POST).

Remediation: install Node from distro packages or verify the NodeSource script signature; pin Bun to a release artifact + sha256sum -c; pin the Archon clone to a signed tag or vendored submodule.

F-INPUT-01 — Unbounded pagination + reflective sort attribute

MEDIUM · Confidence High · OWASP A04/A05 · CWE-770 / CWE-915

Affected: routers/scanner.py (/scanner/results), routers/outcomes.py (/outcomes/signals) — limit without an upper bound; getattr(ScannerEvent, sort_by) resolving a sort column from a raw query param.

Why it matters: an unbounded limit lets one request serialize an entire large table (ScannerEvent, StockAggregate) → memory/CPU DoS. getattr(Model, user_input) widens the surface to unintended attributes.

Remediation: cap limit (the tweets route already does this: Query(50, ge=1, le=200) @tweets.py:59); validate sort_by against an explicit allowlist.

ALLOWED_SORTS = {"detected_at","volume_ratio","price","symbol"}
if sort_by not in ALLOWED_SORTS:
    raise HTTPException(422, "invalid sort field")

F-WS-01 — WebSocket resource exhaustion

MEDIUM · Confidence Medium · OWASP A04 · CWE-770

Affected: routers/live_data.py:20,76, news.py:92, tweets.py:29 — all authenticate via ws_get_current_user (cookie JWT, good) but: no per-IP/user connection cap, no idle/lifetime timeout, no inbound message-size limit, and each connection opens its own Redis pub/sub. @limiter.exempt removes slowapi protection.

Remediation: cap concurrent WS connections per user and globally; add a server-side idle timeout; share one Redis subscription via the existing websocket_manager fan-out; verify the Origin header against CORS_ORIGINS on handshake.

F-CI-01 — Container scan advisory-only; no SAST; CI missing permissions

MEDIUM · Confidence High · GitHub Actions hardening · SLSA · CWE-1104

Affected: .github/workflows/ci-publish.yml:107,133 (continue-on-error: true, Trivy exit-code: "0"); ci.yml (no top-level permissions: → broad default token); no CodeQL/Semgrep.

Why it matters: vulnerabilities are surfaced but never block a merge/publish (verified this session — the #366 publish run's three Trivy scan jobs failed yet the image still shipped); a compromised test step gets a write-capable token.

Remediation: set Trivy exit-code: 1 for HIGH/CRITICAL and drop continue-on-error; add permissions: {contents: read} to ci.yml; add CodeQL + bandit/semgrep; validate inputs.image_tag against ^[a-z0-9._-]+$ in deploy.yml before the SSH shell.

F-DOCKER-01 — docker-socket-proxy allows BUILD/POST; preview compose hardcodes weak creds

MEDIUM · Confidence Medium · CIS Docker 2.x · CWE-798

Affected: docker-compose.yml:424-443 (socket proxy with BUILD/POST); dark-factory/docker-compose.preview.yml (POSTGRES_PASSWORD: preview_password, JWT_SECRET_KEY: preview-only-not-secret-…).

Why it matters: the proxy narrows the socket but still permits container/image creation; the preview compose's hardcoded JWT secret/DB password are fine for ephemeral CI but dangerous if applied to a long-lived host.

Remediation: restrict the proxy to the minimum verbs the factory needs; guard that no preview_* credential appears outside ephemeral CI; verify no *-preview containers persist after teardown.

F-INPUT-02 — Weak / decentralized ticker & free-form field validation

MEDIUM · Confidence Medium · OWASP A03/A04 · CWE-20 / CWE-915

Affected: multiple schemas/* and routers — ticker/symbol inputs validated ad-hoc (.upper()), no central regex/length; some models accept nested Dict[str, Any] without bounds; URL/webhook fields without scheme validation; date-range params without explicit bounds.

Remediation: a shared Ticker constrained type (constr(pattern=r'^[A-Z]{1,6}$')); max_length on string fields; bound date ranges; HttpUrl + https-only for URL fields; model_config = ConfigDict(extra="forbid") on request models to block mass assignment.

F-FRONT-01 — External-content URLs rendered without allowlist; no CSP

LOW · Confidence High · OWASP A03 · CWE-601 / CWE-79

Affected: components/NewsFeed.tsx:154-176 (article_url/image_url/title), ScannerResults.tsx:240-250 (metadata.tweet_url), public/sw.js:38 (notification.data?.url → clients.openWindow), GlobalErrorToast.tsx:149 (rel="noreferrer" missing noopener).

Why it matters: React escapes text, so not stored-XSS today, but unvalidated external href/src and the SW openWindow are open-redirect / javascript:-scheme vectors if upstream data is attacker-influenced; no CSP means a future XSS has no second line of defense.

Remediation: validate external URLs (new URL(), https:-only, domain allowlist); restrict sw.js openWindow to same-origin/relative; add rel="noopener noreferrer"; ship a CSP from Caddy.

F-LOG-01 — Potential sensitive data in logs / Seq

LOW · Confidence Medium · OWASP A09:2021 · CWE-532

Affected: broad logging to Seq (SEQ_URL); database.py:23 echoes SQL when LOG_LEVEL=DEBUG. Prod correctly hides stack traces from HTTP responses (good).

Remediation: add a logging filter redacting known secret keys (*_KEY, *_PASSWORD, DATABASE_URL, access_token); never log settings objects; keep LOG_LEVEL=DEBUG out of production.

F-AUTH-02 — Correction: /api/v1/tweets/recent IS protected

INFO · Confidence High

Affected: routers/tweets.py:57-94. A parallel analysis flagged this as unauthenticated; it is not. The route prefix /api/v1/tweets is not in EXEMPT_PREFIXES, so the global AuthMiddleware requires a valid JWT cookie before the handler runs. Recorded to stop a false-positive propagating. Note: the middleware validates the JWT but does not re-check is_active; routes needing live user state should still depend on get_current_user (which all WS routes do).

Assumptions
