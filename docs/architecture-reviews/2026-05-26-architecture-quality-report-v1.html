Overall Health

MarketHawk is a feature-rich, actively developed full-stack stock scanning platform with two contributors (human + autonomous "Dark Factory" agent). The backend follows a clean layered architecture (routers → services → providers → models) with good separation of concerns. The frontend is a modern React 19 + TypeScript + React Query application with real-time WebSocket support, recently refactored to decompose 5 large page components into focused subdirectories with extracted hooks.

The codebase is functional and well-organized for a solo/small-team project, but has significant gaps in security, testing, and operational readiness that would need to be addressed before any production or multi-user deployment.

Top 5 Strengths

  1. Clean layered architecture — Backend follows consistent router → service → provider → model layering with dependency injection throughout.
  2. Comprehensive domain model — 37 ORM models covering scanners, signals, outcomes, alerts, auto-trading, futures, and journaling; well-indexed with composite constraints.
  3. Provider abstraction — Protocol-based data provider interface (base.py) allows Polygon.io and IBKR to be swapped transparently; IBKR pacing guard is well-engineered.
  4. Autonomous development pipeline — Dark Factory + Archon workflows enable fire-and-forget feature development from GitHub issues to reviewed PRs (117 factory commits, 30% of total).
  5. Structured error tracking — Protocol-based ErrorTrackerFactory with Seq integration, unique error IDs, and frontend-to-Seq deep linking in the GlobalErrorToast.

Top 5 Risks

  1. CRITICAL No authentication/authorization — All 60+ API endpoints are publicly accessible, including auto-trade order submission, system config, and alert management.
  2. CRITICAL CORS wildcard (["*"]) — Hardcoded in config.py:49; not configurable per environment. Combined with no auth, any origin can call any endpoint.
  3. HIGH Zero frontend test coverage — No test framework configured, no test files exist across 86 files. Recent decomposition makes the codebase more testable, but TypeScript strict mode remains disabled.
  4. HIGH No database connection pooling — create_engine() in database.py uses default pool settings; under load this risks connection exhaustion.
  5. HIGH Minimal CI/CD — Single GitHub Actions workflow covers only backend pytest; no frontend checks, no Docker build, no security scanning, no migration validation.

Top 5 Recommended Actions

  1. Add authentication middleware (JWT or API key) to all non-health endpoints — Effort: M, blocks production use.
  2. Make CORS origins environment-configurable and default to restrictive list — Effort: S.
  3. Configure SQLAlchemy connection pooling (pool_size, max_overflow, pool_pre_ping) — Effort: S.
  4. Extend CI to include frontend tsc --noEmit + vite build, Alembic migration check, and dependency audit — Effort: M.
  5. Install Vitest and add tests for the 6 extracted hooks; enable TypeScript strict mode incrementally — Effort: L.

Architectural Pattern

Layered monolith with service extraction in progress. The backend is a monolithic FastAPI application following a classic layered architecture. The live-scanner and tweet-monitor are extracted as separate Docker services, suggesting movement toward a service-oriented architecture. Celery handles async work. The frontend is a standard React SPA communicating via REST + WebSocket.

3. Architecture Assessment

3.1 Separation of Concerns

Score: 4/5 Good

Evidence: Clean four-layer separation: routers handle HTTP, services contain business logic, providers abstract external APIs, models define the domain. Each router depends on services via explicit function calls; services receive DB sessions via dependency injection.

Finding: Some backend routers contain non-trivial business logic (e.g., scanner.py:760 lines, system.py:480 lines), blurring the router/service boundary. The universe.py router was recently refactored to delegate to universe_orchestrator.py (PR #82). On the frontend, a major decomposition refactored the 5 largest page components (AutoTrading 1,003→211 lines, Scanner 830→177, Alerts 812→154, StockDetail, ActiveWatchlist) into subdirectories with focused panel components and extracted hooks — a strong improvement in separation of concerns.

Risk: As features grow, thick routers become harder to test and reuse.

Recommendation: Continue extracting business logic from scanner.py router into dedicated service methods, following the universe refactoring pattern.

3.2 Module Boundaries

Score: 4/5 Good

Evidence: The providers/ package enforces a clear protocol via base.py. Services do not directly import ORM internals. Models are registered centrally in models/__init__.py. The recent task module split (PR #80) broke a god module into domain-focused files.

Finding: Service-to-service imports are present (e.g., scanner.py imports from catalyst_parser, signal_ranker, timeseries_forecast). This is acceptable for a monolith but creates implicit coupling.

Risk: Inter-service coupling could complicate future extraction.

3.3 Dependency Direction

Score: 3/5 Acceptable

Evidence: Generally flows inward: routers → services → models. Providers are consumed by services.

Finding: main.py contains a scan orchestrator registry (lines 188-193) with lazy imports of service modules, creating a top-level coupling point. celery_app.py defines the beat schedule with task names as strings, creating an implicit contract.

Risk: String-based task references can silently break if task modules are renamed.

3.4 Coupling and Cohesion

Score: 3/5 Acceptable

Evidence: Most services are cohesive (one domain per module). The scanning.py task file orchestrates alerts + auto-trading after each scan, coupling scan execution with downstream effects.

Finding: The alert_service.py (401 lines) handles rule evaluation, four delivery channels, delivery logging, and cooldown tracking — high cohesion around alerting but could benefit from channel-specific extractors.

3.5 API Design

Score: 4/5 Good

Evidence: Consistent REST conventions: POST for creation (201), 202 for async operations, 409 for concurrency conflicts. All endpoints use Pydantic schemas for request/response validation. WebSocket endpoints for real-time scan progress and live market data.

Finding: The columnar-compact data format for historical bars (stocks.py router) is a thoughtful optimization, mapping column names to 1-2 character keys for bandwidth reduction.

Risk: No API versioning. Breaking changes would affect all clients simultaneously.

3.6 Data Access Patterns

Score: 3/5 Acceptable

Evidence: SQLAlchemy ORM used consistently. Transactions managed via session scope in get_db() dependency. SAVEPOINT-based test isolation in conftest.py is well-engineered.

Finding: No eager loading (joinedload) observed in router queries. The ScannerEvent.latest_review property computes O(n) over the reviews list instead of using a database query. JSONB columns (indicators, metadata, criteria) are used for flexible payloads but are harder to query and index.

Risk: N+1 query patterns likely exist for endpoints that traverse relationships.

Recommendation: Audit top-traffic endpoints with SQL logging enabled and add joinedload/selectinload where relationships are always needed.

3.7 Error Handling Architecture

Score: 4/5 Good

Evidence: Custom MarketHawkError exception hierarchy with is_retryable flag. Global exception handler in main.py generates unique error IDs via MD5 hash of traceback. Stack traces hidden in production (ENVIRONMENT != development). Seq integration for centralized error collection.

Finding: Domain-specific errors (UniverseNotFoundError, UniverseValidationError) added in recent refactoring, moving away from generic HTTPException in routers.

3.8 Configuration Management

Score: 2/5 Weak

Evidence: config.py uses raw os.getenv() calls instead of Pydantic BaseSettings. No type validation or coercion on environment variables. IBKR_PORT and IBKR_CLIENT_ID do bare int() conversion that would crash with a non-numeric value.

Finding: The docstring says "pydantic-settings" but the class doesn't inherit from BaseSettings. CORS origins are hardcoded (["*"]) rather than environment-configurable. ENVIRONMENT defaults to "production" — a safe default.

Risk: Invalid env vars cause unhandled ValueError on startup instead of clear validation errors.

Recommendation: Migrate to pydantic-settings.BaseSettings for automatic env validation, type coercion, and .env file support.

3.9 Observability Architecture

Score: 3/5 Acceptable

Evidence: Seq for structured logging with error IDs. Flower for Celery monitoring. pgAdmin for database inspection. Frontend GlobalErrorToast includes Seq deep links.

Finding: No metrics (Prometheus/StatsD), no distributed tracing, no APM. System health endpoint checks DB, Redis, and IBKR connectivity but doesn't expose metrics for scraping.

Risk: Performance degradation and capacity issues would be invisible until they cause failures.

3.10 Security Architecture

Score: 1/5 Poor

Evidence: No authentication or authorization layer detected across any of the 15 routers and 60+ endpoints. CORS allows all origins. No rate limiting middleware. No CSRF protection.

Finding: The IB Gateway runs with READ_ONLY_API=yes by default, which is a good safety measure. However, the auto-trading router (auto_trading.py) submits orders to IBKR with no auth gate. The system/config endpoint allows arbitrary system configuration updates.

Risk: Any network-accessible client can submit trades, modify system configuration, and access all user data.

Recommendation: This is the #1 priority. Add JWT or API key authentication before any network exposure.

3.11 Deployment Architecture

Score: 3/5 Acceptable

Evidence: Docker Compose defines 13 services with health checks, dependency ordering, and network segmentation. External volumes for data persistence. Read-only bind mounts for backend code. Optional profiles for forecast-worker, dark-factory, and backlog-scheduler.

Finding: No memory/CPU limits on most services. Management tools (pgAdmin, Flower, Seq) exposed on all interfaces. No TLS termination. No production vs. development compose file separation.

Risk: Resource-hungry services (ML forecasting, data sync) could starve critical services (API, database).

8. Performance & Scalability

Performance Findings

Strengths

  • Columnar data format — Historical bar data uses a compact column-oriented format with 1-2 char keys, reducing JSON payload by ~60%.
  • GZip middleware — Compresses responses >1KB.
  • Celery offloading — Heavy scans, data sync, and ML analysis run as background tasks.
  • IBKR pacing guard — Sliding-window rate limiter prevents API bans (55 req/10min, 2s minimum gap).
  • Database indexes — Composite indexes on frequently queried columns (ticker+timestamp, ticker+event_date+scanner_type).

Bottlenecks & Risks

  • No connection pooling — database.py calls create_engine() with no pool_size/max_overflow. SQLAlchemy defaults to pool_size=5, which is low for concurrent API + Celery workers.
  • Synchronous ORM — Despite having asyncpg in requirements, the app uses synchronous SQLAlchemy. Each request blocks a thread during DB access.
  • Frontend polling — System status polled every 30s, alert stats every 30s, universe sync status every 5s, scan status every 5s during active scans. Multiple timers could strain the API under many tabs.
  • Discovery service delays — 15-second hard-coded delays between batch Polygon requests. No dynamic backoff.
  • No caching layer — Redis is used as a Celery broker but not for application-level caching (API response caching, query result caching).
  • Forecast worker — Requires 36GB RAM for a 200M-parameter model. No horizontal scaling beyond single worker.

Scalability Model

Horizontal: Celery workers can be scaled by adding containers. The API server runs a single instance; scaling would require session-affinity for WebSocket connections. Database is a single PostgreSQL instance with no read replicas.

Vertical: The forecast worker explicitly requires 32GB+ RAM. Other services have no memory limits configured.

Verdict: Adequate for single-user/small-team workloads. Would need connection pooling, async ORM, and read replicas for multi-user production.

9. API, Data & Integration Quality

API Consistency

Score: 4/5. Endpoints follow consistent naming (/api/{domain}/{action}). HTTP status codes are appropriate (201, 202, 409). All endpoints use Pydantic schemas. FastAPI auto-generates OpenAPI docs at /docs.

Gap: No API versioning (/api/v1/). Error response format varies slightly between router-thrown HTTPExceptions and the global MarketHawkError handler.

Data Model Clarity

Score: 4/5. 37 models with clear naming. Relationships defined via ForeignKey + relationship(). Composite unique constraints prevent duplicate scanner events. UUID fields provide external-safe identifiers alongside integer PKs.

Gap: Timestamps stored as naive UTC (datetime.now(timezone.utc).replace(tzinfo=None)) — an intentional choice but could confuse future contributors. JSONB columns lack schema validation.

External Integration Patterns

Polygon.io: Wrapped in massive.py with lazy initialization, auto-pagination, and timestamp conversion. Rate limits handled by the SDK. Well-integrated.

IBKR: Wrapped in ibkr.py + ibkr_orders.py with pacing guard, contract mapping, and duration chunking. Separate client IDs for API (10), trading (11), and live scanner (5). Well-integrated, sophisticated.

Seq: HTTP POST to ingestion endpoint with structured JSON payload. Graceful fallback to stdout. Clean integration.

Tweet Monitor: Separate microservice with its own Dockerfile, scraping X.com with auth cookies. Requires manual token rotation every 30 days. Fragile dependency on cookie-based auth.

Migration Approach

Score: 3/5. 37 Alembic migrations auto-generated from model changes. Migrations not tested in CI. No rollback documentation. No zero-downtime migration strategy for large tables.

15. Final Recommendations

Most Important Architectural Decision to Make Next

Authentication strategy. Every other improvement is secondary while 60+ endpoints — including trade execution and system configuration — are accessible without any credentials. Choose between JWT tokens (standard for SPAs), API keys (simpler, adequate for single-user), or OAuth2 (if multi-user is planned). This decision shapes middleware, frontend auth flow, and Celery task authentication.

Most Important Engineering Practice to Improve Next

CI/CD pipeline expansion. The current single-workflow CI (backend pytest only) is inadequate for a 45,000-line, 13-service codebase. Adding frontend type checking + build, migration validation, and dependency scanning would catch the majority of regressions before merge. This is a force multiplier — especially with 30% of commits coming from the autonomous Dark Factory agent.

Most Important Testing Gap to Close Next

Frontend test coverage. 15,100 lines of TypeScript across 86 files with zero tests is the single largest quality risk. The recent page-decomposition refactoring (Scanner, Alerts, AutoTrading, StockDetail, and ActiveWatchlist split into focused sub-components) has made the codebase significantly more testable — the 6 custom hooks (useScannerState, useScannerWs, useWatchlistLive, useLiveStockData, useScanTask, useScorecard) contain state and WebSocket logic that can be unit-tested independently of rendering. Start with Vitest + React Testing Library for hooks and the GlobalErrorToast.

Most Important Operational Risk to Reduce Next

Automated database backups. All scanner events, trade journals, signal reviews, alert rules, and system configuration live in a single PostgreSQL instance with no automated backup. A volume corruption or accidental docker volume rm would cause total data loss. A simple cron + pg_dump with retention policy costs hours to implement and eliminates an existential risk.

Conclusion

MarketHawk is a well-architected application for its stage of development — a solo/small-team tool with impressive breadth (5 scanner types, auto-trading, ML signal ranking, futures, alerting, journaling). The codebase is clean, consistently organized, and actively improved through both human and AI-driven development.

The primary concern is that the system has grown beyond its original "local tool" scope (particularly with auto-trading and multi-channel alerts) without a corresponding investment in security, operational readiness, and testing rigor. The gap between feature ambition and infrastructure maturity is the central risk.

Addressing the top 5 immediate actions — authentication, CORS, connection pooling, port binding, and rate limiting — would significantly reduce risk within 2 weeks of focused effort. The longer-term improvements (CI expansion, frontend testing, metrics, async migration) represent a path toward production-grade maturity.
