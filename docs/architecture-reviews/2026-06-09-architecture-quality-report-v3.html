Overall Health

In the six days since v2, MarketHawk closed 15 of 17 round-2 tickets (#190–#205, label architecture-audit-v2) and — unlike most teams — the fixes are real: the empty JWT signing key now fails startup via a field_validator (config.py:137-145), all WebSocket routes authenticate before accept() (core/auth.py:84-99), CSRF and login brute-force limits are enforced, the frontend tsc gate is verifiably green (exit 0, run during this review), worker Prometheus metrics flow through a shared multiproc volume, and circuit breakers + outbound timeouts landed with a 256-line test suite (PR #248). The defining v2 maintainability defect was also closed: services/scanner.py (1,033 lines) and services/futures_data.py (1,086) are now thin facades over ten focused submodules, with every prior circular import eliminated.

The character of the risk has shifted again. v1 was missing foundations; v2 was unfinished edges; v3's risk is process honesty at autonomous speed: one closed ticket shipped no artifacts at all (#195 automated backups — the repo still has only a manual pg_dump), ruff is red on main with 65 errors that no CI gate catches, decomposition moved complexity without always reducing it (run_screen is still CC≈43), and the worker-metrics fix is verified by code reading, not by observing a live scrape. With 66% of commits now authored by the factory, the gates — not the reviewers — are the quality system, and two of them have holes.

Top 5 Strengths (new since v2)

  1. God-module decomposition, verified — scanner.py 1,033→206 and futures_data.py 1,086→90, logic relocated into pre_market_scan (387), oversold_bounce_scan (237), session_metrics (144), scan_enrichment (217), and four futures services (138–473 each); originals are backward-compatible facades; dependency graph fully acyclic.
  2. Security finish-work complete — all five v2 sharp edges closed and verified in code: JWT key validator, pre-accept WebSocket auth on all 3 WS routes, X-Requested-With CSRF middleware on mutating routes, 5/min auth rate limits, secure cookies default-on with an optional Caddy TLS profile (HSTS, HTTP→HTTPS).
  3. Resilience toolkit landed — pybreaker circuit breakers on Polygon (5 fails/60s) and IBKR (3/120s) mapped into the typed exception hierarchy, explicit Polygon connect/read timeouts, and HTTPXClientInstrumentor closing the v2 outbound-tracing gap.
  4. CI gates now honest — tsc --noEmit exit 0 (verified), ESLint TS ruleset actually loads (spread bug fixed, eslint.config.js:44-46), pip-audit/npm-audit blocking, frontend coverage measured app-wide with thresholds set from measured reality (~21%) instead of a hand-pinned 7-file include.
  5. Container hardening — direct Docker socket mounts replaced by a restricted socket-proxy (EXEC/AUTH/SECRETS blocked), backend and factory run as non-root users, management of the multiproc dir handled via Dockerfile chown.

Top 5 Risks

  1. HIGH Backups ticket closed with no implementation — #195 is closed, but the repo contains no backup service, cron, script, or workflow; only the manual pg_dump line in deployment-guide.md. The v1-era existential data-loss risk persists, now hidden behind a green ticket. (R01)
  2. MEDIUM ruff check is red on main — 65 errors (verified, exit 1); backend lint runs in pre-commit but not CI, and factory commits bypass pre-commit (known spillover, #275/#276). A documented gate that doesn't gate. (R02)
  3. MEDIUM New duplication epidemic — .astimezone(timezone.utc).replace(tzinfo=None) now appears ~96 times (the older naive-UTC pattern improved 87→53 files, but its cousin grew unchecked); still no to_utc_naive()/utc_now() helper, no get_or_404 (404 pattern 26→28). (R03)
  4. MEDIUM Complexity moved, not reduced — DiscoveryService.run_screen remains the worst function (217 lines, CC≈43, untouched across three reviews); run_pre_market_scan is a 346-line CC≈37 function inside its new module. (R04)
  5. MEDIUM Deploy-path gaps — /api/health is still liveness-only (no DB/Redis readiness), and migrations don't auto-run on container start; a rolling update with a pending migration can serve traffic against a stale schema. (R05)

Top 5 Recommended Actions

  1. Reopen #195 and implement real backups: pg_dump sidecar on cron + retention + a restore smoke-test — Effort: M, closes R01 and a three-review-old High.
  2. Add ruff check to the CI backend job and run ruff --fix once (all 65 are auto-fixable) — Effort: S, closes R02.
  3. Create app/utils/time.py (utc_now(), to_utc_naive()) and codemod the ~149 combined call sites; add get_or_404() for the 28 router instances — Effort: M, closes R03.
  4. Decompose run_screen into per-asset-class screeners behind one interface — Effort: M, closes the CC≈43 hotspot (R04).
  5. Add a readiness endpoint (DB/Redis probes) and an explicit migration gate in the deploy path — Effort: S–M, closes R05.

3. Architecture Assessment

3.1 Separation of Concerns

Score: 4/5 Good Δ 0

Change: The v2 finding — "router-thinning relocated complexity into the service layer" — is resolved: the absorbing god module was decomposed (#199). Routers stayed thin (system.py 140, scanner.py 708).

Finding (caps the score): alerts.py (21 db.query calls) and auto_trading.py (20) still have no service layer and keep hand-rolled _rule_to_dict/_strategy_to_dict serializers — CRUD logic and response shaping live in the HTTP layer for two domains.

3.2 Module Boundaries

Score: 4/5 Good Δ 0

Change: Ten new focused submodules with clean seams; facades preserve the old import surface; futures domain split by responsibility (contracts / aggregates / rollovers / series).

Finding (caps the score): the v2 cross-layer leak persists — services/universe_export.py:51,59 still raises HTTPException from the service layer (defensible as input validation, but it is the one place a web type crosses the seam).

3.3 Dependency Direction

Score: 3/5 Acceptable Δ 0

Change: All circular imports eliminated — scanner↔alert_service and stock_data↔futures_data are now method-level lazy imports (scanner.py:146, stock_data.py:247); no service imports a router (verified).

Finding (caps the score): the registry is still populated by side-effect importlib imports in main.py:428-434 and the beat schedule still references tasks by string name — both work, but renames break silently at runtime, not import time.

3.4 Coupling and Cohesion

Score: 4/5 Good Δ +1

Change: The defining improvement of v3. Each scanner strategy is its own cohesive module registering a descriptor; futures concerns are separated; session_metrics and scan_enrichment extract the shared logic that v1 flagged as duplicated inside the monolith. Dependency graph acyclic end-to-end.

Finding (caps the score): cohesion inside the new modules is uneven — pre_market_scan.py is one 346-line function doing detection + enrichment + persistence in a single loop; liquidity_hunt.py (708) remains the largest strategy.

3.5 API Design

Score: 5/5 Excellent Δ 0

Finding: No drift. All 13 app routers under /api/v1; auth (/api/auth) and health (/api) intentionally unversioned per ADR-0010; frontend mirrors via versioned + unversioned clients; an ESLint rule now forbids raw /api/ strings outside the api layer (eslint.config.js:58-76) — the convention is machine-enforced.

3.6 Data Access Patterns

Score: 4/5 Good Δ 0

Finding (caps the score): Eager loading remains confined to two files (routers/scanner.py:376, journal_service.py:20-22); the v2-flagged joinedload-on-one-to-many with limit/offset pagination pattern is still present; journal_service.get_trade_stats() loads every Trade and aggregates in Python instead of SQL. JSONB event columns remain schema-free on write.

3.7 Error Handling Architecture

Score: 5/5 Excellent Δ 0

Change: The typed hierarchy gained a productive consumer: CircuitBreakerError maps to ProviderError(is_retryable=False), so an open breaker correctly suppresses Celery retries instead of hammering a failing provider. Structured context flows to Seq; deterministic error IDs unchanged.

3.8 Configuration Management

Score: 5/5 Excellent Δ +1

Change: The v2 cap is gone: JWT_SECRET_KEY now has a field_validator rejecting keys <32 chars with a generation hint (config.py:137-145), and is a required field with no default. Breaker/timeout/pool settings are all typed, validated, env-tunable. No raw os.getenv anywhere in app code (verified).

3.9 Observability Architecture

Score: 5/5 Excellent Δ +1

Change: Both v2 caps closed. Workers write to a shared prometheus_multiproc volume (compose lines 132/203) owned by appuser (Dockerfile 23-24), aggregated by MultiProcessCollector on the scraped /metrics endpoint (main.py:399-409), with mark_process_dead cleanup on worker shutdown. Outbound HTTP is traced via HTTPXClientInstrumentor; trace IDs flow into Seq logs.

Finding: pipeline correctness is verified statically (code + config), not by observing a live scrape — see Limitations. Live-scanner process metrics remain the one unexported corner.

3.10 Security Architecture

Score: 4/5 Good Δ +1

Change: Every v2 sharp edge verified closed: key validation (#190), pre-accept WS auth on all 3 routes (#191), CSRF header middleware exempting only /api/auth/* (#192), 5/min limits on login/register/refresh (#196), secure cookies default-on + SameSite=Strict + Caddy TLS profile (#202), socket-proxy + non-root containers (#203).

Finding (caps the score): no password policy on register, authorization is still authenticated-vs-not (no roles — deliberate single-user design with registration closing after first user), Trivy image scans are continue-on-error, no audit trail for sensitive actions, and the default compose still sets ENVIRONMENT=development (traces exposed on a default stack).

3.11 Deployment Architecture

Score: 4/5 Good Δ 0

Change: TLS termination now exists (Caddy, opt-in profile), Docker socket exposure is proxied and scoped, app containers run non-root.

Finding (caps the score): TLS is an optional profile rather than the default path; migrations are not part of the deploy path (manual alembic upgrade); /api/health gives a green liveness signal before DB/Redis are usable — together these leave a deployment-ordering trap.

8. Performance & Scalability

  • Provider failure cost — open breakers fail fast (no thread-blocking on a dead Polygon/IBKR), explicit Polygon timeouts bound worst-case latency.

Still open

  • Sync ORM (ADR-0004, by design) — requests block threads on DB access.

Scalability model: unchanged from v2 — Celery scales horizontally; API is single-instance with WS affinity unsolved; one Postgres, no replicas. Adequate for the single-user design point; the new breakers make provider-side degradation graceful rather than cascading.

18. Final Recommendations

Most important decision to make next

Decide what "done" means in an autonomous pipeline. #195 closed with zero artifacts — the second fictional completion in two reviews (#107 was the first). With the factory authoring 66% of commits, ticket state is no longer evidence. Add a verification step to the factory's close path (the validate stage already exists) or adopt the rule that risk-register items close only on artifact-verified review.

Most important practice to improve next

Put every documented gate where the work actually happens. The repo's stated quality system includes ruff — but ruff runs only in pre-commit, which factory commits bypass, so main sits at 65 lint errors while CI shows green. One CI step closes the gap; the deeper habit is auditing each gate for "can the primary author bypass this?"

Most important operational risk to reduce next

Backups, for the third consecutive review. Every scanner event, review verdict, trade journal, and the users table live in one Postgres volume with no automation. This is now the single largest gap between the 90/100 score and reality — hours of work, existential downside, and a ticket history that falsely says it's handled.

Conclusion

v3 confirms the pattern v2 established: this codebase converts findings into shipped fixes at remarkable speed — 15 of 17 round-2 tickets genuinely closed in six days, including the marquee decomposition that v1 first asked for. Overall quality rose 83→90, architecture 3.9→4.3, and for the first time no dimension sits below 3.

The trajectory's new constraint is trust calibration. The system now improves itself faster than a human can audit it, and the two defects that matter most in v3 — a fictional ticket closure and a bypassed lint gate — are both failures of verification, not of engineering. Harden the close-the-loop machinery (artifact-verified ticket closure, gates in CI rather than pre-commit, runtime confirmation of statically-verified fixes) and the score takes care of itself.
