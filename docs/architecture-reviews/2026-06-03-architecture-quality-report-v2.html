Overall Health

In the eight days since v1, MarketHawk closed 22 of 24 remediation tickets (#84–#107) generated directly from the first report. The two structural weaknesses that defined v1 — a complete absence of authentication and no operational instrumentation — have been genuinely addressed: cookie-based JWT auth now gates the HTTP surface, and a full Prometheus + Grafana + OpenTelemetry/Jaeger observability stack is wired in. Configuration moved to validated pydantic-settings, the database now pools connections, Redis caches hot endpoints, the API is versioned under /api/v1, and the four previously-missing docs plus eleven ADRs now exist.

The quality score rose from 62 to 83 and the codebase is now credibly approaching production-readiness. The remaining risk is no longer "features outpacing infrastructure" but finish-quality on the new infrastructure: an empty default signing key, an auth layer that doesn't cover WebSockets, a metrics pipeline that silently drops worker metrics, and a frontend typecheck that is currently red. These are sharp, fixable edges rather than missing foundations.

Top 5 Strengths (new since v1)

  1. Authentication & access hardening — JWT via HttpOnly cookies with server-side refresh-token rotation in Redis (core/auth.py, routers/auth.py), CORS narrowed to an env-driven allowlist, slowapi rate limiting enabled by default, and all management ports (pgAdmin/Flower/Seq) rebound to 127.0.0.1 with Flower basic-auth.
  2. Full observability stack — 12 custom Prometheus metrics + FastAPI/Celery/Polygon/IBKR instrumentation, four provisioned Grafana dashboards with alerting rules, OpenTelemetry tracing into Jaeger, and a GELF sidecar for container log capture.
  3. Validated configuration & pooled data access — Settings(BaseSettings) with field validators replaces raw os.getenv; create_engine now configures pool_size/max_overflow/pool_pre_ping (all env-tunable and exported as gauges).
  4. API versioning + error hierarchy — every application route now lives under /api/v1 (ADR-0010), and a typed MarketHawkError hierarchy in exceptions.py drives retry semantics and deterministic error IDs.
  5. Documentation & decision capture — DEVELOPMENT / ARCHITECTURE / ENV_VARIABLES / POLYGON_RATE_LIMITS docs all created and substantive; ADRs 0002–0010 record the auth, sync-ORM, JSONB, Celery, live-scanner, dark-factory, naive-UTC, and versioning decisions the v1 report asked for.

Top 5 Risks (v2 — mostly newly introduced)

  1. CRITICAL Empty default JWT signing key, no validator — JWT_SECRET_KEY: str = "" (config.py:53) has no field_validator (unlike DATABASE_URL/ports). A deploy that forgets to set it signs every token with the empty string — tokens become trivially forgeable. auth.py:31 signs with it; no startup guard.
  2. HIGH WebSocket endpoints bypass authentication — the ASGI auth middleware returns early for any non-HTTP scope (main.py:251–253); the ~9 WS routes accept() with no token check, so live ticks, scan progress, system state, and tweet signals stream unauthenticated.
  3. HIGH Frontend typecheck / build is red — tsc --noEmit exits 2 (verified): tsconfig.json includes src with no test exclusion or runner types, and useScannerWs.test.ts:19 has a genuine strict implicit-any. The same command gates npm run build and CI.
  4. HIGH Worker metrics never reach Prometheus — Celery/live-scanner write multiprocess metrics to a per-container tmpfs volume and Prometheus only scrapes backend:8000, so celery_tasks_total et al. are recorded but invisible; the Celery dashboard and "high failure rate" alert have no data.
  5. MEDIUM No CSRF protection on cookie auth + several gaps — cookie auth with no CSRF token on state-changing routes (incl. order submission); login/register have no brute-force limit; secure cookie flag is off in the default development config; ESLint's TS ruleset silently doesn't load so any is unenforced.

Top 5 Recommended Actions

  1. Add a field_validator rejecting an empty/short JWT_SECRET_KEY at startup; rotate any key used in a real deploy — Effort: S, closes the lone CRITICAL.
  2. Authenticate WebSocket handshakes (validate the access_token cookie before accept()) and add a CSRF defense (double-submit token or required custom header) on mutating routes — Effort: M.
  3. Exclude test files from the production tsc (or add vitest/globals + jest-dom types and fix the implicit-any); make the CI frontend job genuinely green — Effort: S.
  4. Fix the metrics export path (shared multiproc dir or per-process scrape targets) so worker metrics surface; fix eslint.config.js:45 so the TS ruleset loads — Effort: S–M.
  5. Implement the still-missing v1 items that never got a ticket: automated DB backups, and decompose the scanner.py/futures_data.py god modules — Effort: M.

Architectural Pattern

Layered monolith, now hardened with cross-cutting middleware. The router → service → provider → model layering is intact. The new auth, metrics, and tracing concerns are implemented as ASGI middleware composed in main.py's create_app(). Live-scanner and tweet-monitor remain extracted services; the system is now genuinely multi-service (20 compose services including Prometheus, Grafana, Jaeger, and the GELF sidecar).

3. Architecture Assessment

3.1 Separation of Concerns

Score: 4/5 Good Δ 0

Change: #96 thinned the worst routers — routers/system.py fell 480→133 lines (delegates to SystemService), routers/scanner.py 857→698 (extracted ScannerQueryService + scan_orchestrator). Real wins at the router boundary.

Finding: Extraction was incomplete and pushed logic into an already-oversized service — services/scanner.py grew 804→1,033 lines. Config CRUD (routers/system.py:75-89) and the results query (routers/scanner.py:370) remain inline. Net: cleaner routers, worse service-layer locality. No net score change.

3.2 Module Boundaries

Score: 4/5 Good Δ 0

Change: New focused services (scanner_query_service, system_service, outcome_service, journal_service) sharpen boundaries; the models layer has zero upward dependencies.

Finding: One new cross-layer leak — services/universe_export.py:51,59 raises HTTPException (a web concern) from the service layer, the exact thing scan_orchestrator.py:91 documents avoiding.

3.3 Dependency Direction

Score: 3/5 Acceptable Δ 0

Finding: Unchanged. main.py:391-397 still populates the scan-orchestrator registry via side-effect importlib imports; celery_app.py still references beat tasks by string name (tasks now carry explicit name= decorators, so strings at least match declarations).

3.4 Coupling and Cohesion

Score: 3/5 Acceptable Δ 0

Change: Alert evaluation and auto-trade are now separate Celery tasks chained via execute_auto_trade.delay() (tasks/scanning.py:70-73) rather than inlined — a marginal decoupling. alert_service.py was not decomposed (401→417 lines).

3.5 API Design

Score: 5/5 Excellent Δ +1

Change: #105 closed the lone v1 gap. Every application router carries a /api/v1/… prefix; auth (/api/auth) and health (/api) are deliberately excluded per ADR-0010. The frontend mirrors it (VITE_API_BASE_URL ?? '/api/v1' with a separate unversionedClient); WebSockets are versioned too. Strongest single architectural improvement.

3.6 Data Access Patterns

Score: 4/5 Good Δ +1

Change: #100 made ScannerEvent.latest_review O(1) (relationship ordered by reviewed_at desc), added joinedload on scanner results and selectinload in journal_service.

Finding: Eager-loading coverage is thin (only two files), so N+1 risk remains elsewhere; and joinedload on a one-to-many combined with SQL limit/offset (routers/scanner.py:363,370) can mis-paginate — selectinload would be safer.

3.7 Error Handling Architecture

Score: 5/5 Excellent Δ +1

Change: #64 standardized conventions. app/exceptions.py defines a typed hierarchy (MarketHawkError + ScanError, DataFetchError, ProviderError, UniverseNotFoundError/ValidationError) with an is_retryable flag that drives both Celery retries and a dedicated FastAPI handler (retryable→503, else 422). The global handler still emits deterministic ERR-<hash> IDs and gates stack-trace exposure on ENVIRONMENT.

3.8 Configuration Management

Score: 4/5 Good Δ +2

Change: #91 fully delivered. Settings(BaseSettings) with SettingsConfigDict(env_file=".env"); required fields (DATABASE_URL, POLYGON_API_KEY) fail fast at import; field_validators enforce the DB URL scheme, port ranges, and normalize log level/env. Cached via @lru_cache. Raw os.getenv for config is gone.

Finding (caps the score): the single most security-critical field — JWT_SECRET_KEY — defaults to "" with no validator (config.py:53). The framework that makes validation trivial is in place but unused on the field that needs it most. (See R01.)

3.9 Observability Architecture

Score: 4/5 Good Δ +1

Change: From "no metrics, no tracing, no APM" to a complete stack: 12 Prometheus metrics (core/metrics.py), an ASGI instrumentation middleware, OpenTelemetry → Jaeger (core/tracing.py, fail-open), four provisioned Grafana dashboards with alerting rules (IBKR-disconnected, high Celery failure rate), and a GELF sidecar.

Finding (caps the score): the export path is broken for non-backend processes — Celery/live-scanner write to a per-container tmpfs multiproc volume and Prometheus only scrapes backend:8000, so worker metrics never surface and the Celery dashboard/alert have no data. Outbound Polygon/httpx calls are also not traced. (See R06.)

3.10 Security Architecture

Score: 3/5 Acceptable Δ +2

Change: The defining v1 weakness is largely closed. A pure-ASGI AuthMiddleware enforces JWT on the HTTP surface (main.py:246-280), with a well-built refresh-token flow (opaque random tokens stored in Redis, rotated on use, revoked on logout). CORS is an env allowlist; rate limiting is on; management ports are loopback-bound.

Finding (caps the score): the auth layer has sharp gaps — empty default signing key (R01), WebSocket bypass (R02), no CSRF (R03), no login brute-force limit, root-running containers, and the docker.sock mount persists. Authorization is still authenticated-vs-not only (no roles on User).

3.11 Deployment Architecture

Score: 4/5 Good Δ +1

Change: #104 + #93 + #146 added a GHCR image registry (all app images tagged), three CI/CD workflows (PR gate, publish + Trivy scan, manual SSH deploy), memory limits on 15 of 20 services, and a clean dev/live split via docker-compose.override.yml.

Finding: still no TLS termination; the deploy workflow is scaffolded (host/SSH secrets unprovisioned); five services (frontend, prometheus, grafana, jaeger, dark-factory) have no resource limits; Prometheus/Grafana/backend/frontend bind to 0.0.0.0.

8. Performance & Scalability

Resolved since v1

  • Connection pooling (#85) — pool_size=20, max_overflow=10, pool_pre_ping=True, env-tunable.
  • Application caching (#98) — core/cache.py read-through over 6 hot endpoints (scanner types/configs, universe list, stock details, system storage/status) with TTLs and explicit invalidation on mutation; fail-open if Redis is down.
  • Eager loading (#100) — latest_review now O(1); joinedload/selectinload on hot reads.
  • Frontend code splitting (#102) — all 14 routes are React.lazy behind a single Suspense boundary.

Still open

  • Synchronous ORM — intentionally retained (ADR-0004); each request still blocks a thread during DB access.
  • 15-second discovery delays — discovery_service.py:21 still hard-codes a 15s gap between Polygon batches (free-tier throttle).
  • Dashboard polling — non-scan views still poll on intervals; only scan/live use WebSockets.
  • Single API instance / single DB — pooling + caching reduce per-instance load, but no read replicas or WS session affinity.

15. Final Recommendations

Most important decision to make next

Treat the auth layer as 80% done, not done. The hard part (cookie JWT, refresh rotation, middleware) is built and good. But an empty default signing key, an unauthenticated WebSocket surface, and missing CSRF mean the system is not yet safe to expose. A focused day closes all three.

Most important practice to improve next

Make CI honestly green. A red tsc gate and a silently-disabled ESLint ruleset mean the expanded CI (#88/#92/#97) is providing less assurance than its dashboard suggests — especially with ~30% of commits from the autonomous factory. Fix the gates so they fail loudly on real problems.

Most important operational risk to reduce next

Automated database backups. This was a v1 High that never received a ticket and remains at 1/5. Every scanner event, trade journal, alert rule, and the new users table lives in one Postgres volume with no automated backup. Hours of work; eliminates an existential risk.

Conclusion

MarketHawk did exactly what a healthy team does with a quality report: it turned the findings into tickets and executed them, lifting the overall score from 62 to 83 and architecture quality from 3.2 to 3.9 in eight days. Authentication, observability, configuration, caching, versioning, pooling, and documentation all moved from "absent" to "present and largely well-built."
