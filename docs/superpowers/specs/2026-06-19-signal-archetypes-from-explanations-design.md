# Signal Archetypes from Explanation Traits and Outcomes — Design

**Date:** 2026-06-19
**Issue:** #466 (Epic: #449 Explanation-Aware Edge Intelligence)
**Status:** Pending review
**Blocked by:** #463 (Extract analysis-ready features from scanner explanations), #465 (Add explanation trait performance aggregation)

## Overview

Signal archetypes are named cluster profiles that group past scanner events by their explanation traits and observed outcomes. The goal is to move beyond raw indicator clusters (which say "high vol_ratio + low rsi") to explanation-aware archetypes (which say "all criteria passed, high confidence" or "volume criterion high + data quality warning") that describe *why* a group of signals fired and how they have historically resolved.

This issue extends the existing `signal_clusters`/`SignalAnalysisRun` infrastructure in `backend/app/tasks/quality.py` to add a second, independent K-means clustering pass that uses explanation-derived features from #463 instead of raw indicator values. The two passes produce separate, non-overlapping cluster assignments on `ScannerEvent`.

## Requirements (from brainstorming)

### Functional
1. A new Celery task `generate_signal_archetypes` clusters events using explanation features (criterion observed values, pass/fail flags, warning flags, confidence score) — not raw indicator values.
2. The feature matrix input comes from the `extract_explanation_features()` function provided by blocker issue #463. The archetype task calls this function and expects a `pd.DataFrame` with columns `event_id`, `interval_key`, `pct_change` (from `ScannerOutcomeSnapshot`), plus explanation-derived feature columns.
3. Outcome summaries (`ScannerOutcomeSummary` metrics) feed the per-cluster `return_profile`, exactly as in the existing `analyze_signal_features` task — they are NOT clustering inputs (to avoid target leakage).
4. The archetype task can be filtered by `scanner_type` (nullable, same as existing task).
5. The minimum event gate is **100 events** with a non-null `explanation` column AND completed outcome snapshots (`ScannerOutcomeSummary.is_complete = True`). Below 100, the run is marked `status="failed"` with a descriptive `error_message`.
6. Per-cluster: clusters with `event_count < 30` receive a `"low_sample": true` flag at the top level of their `return_profile` JSONB. This is advisory — small clusters are persisted and assigned to events but the flag signals unreliable return statistics.
7. Each archetype gets a human-readable label generated by a new `generate_explanation_label()` method on `StatisticalDiscoveryService`, which translates explanation feature names (criterion IDs, warning codes) to plain English. Format mirrors the existing `generate_label` (top-2 features by centroid deviation from global mean, joined with ` + `).
8. Archetype cluster assignments are stored on a new nullable FK column `archetype_cluster_id` on `ScannerEvent` — separate from `signal_cluster_id` (which preserves indicator-based assignments from `analyze_signal_features`). This prevents the two tasks from overwriting each other.

### Schema
9. `SignalAnalysisRun` gains a new nullable `run_type` column (`String(50)`) to distinguish run kinds. Values: `"indicator"` (set by the existing `analyze_signal_features` task) and `"explanation_archetype"` (set by the new task). Backward-compatible: existing rows remain `null` (treated as `"indicator"`).
10. `ScannerEvent` gains a new nullable `archetype_cluster_id` FK → `signal_clusters.id` with an index.

### API
11. `POST /api/v1/outcomes/archetypes` — triggers `generate_signal_archetypes.delay(scanner_type, k)`. Returns `AnalysisTriggerResponse(task_id=...)`.
12. `GET /api/v1/outcomes/archetypes/latest?scanner_type=...` — returns latest completed `explanation_archetype` run and its clusters (same `LatestAnalysisResponse` shape as the existing `/analysis/latest`). Clusters include `low_sample` from their return_profile.

### Beat schedule
13. New Celery beat entry: runs `generate_signal_archetypes` at **11:30 UTC weekdays** (30 minutes after `analyze_signal_features` at 11:00 UTC, ensuring indicator runs complete first).

### Tests
14. Unit tests cover three buckets per the acceptance criteria:
    - **Deterministic labels**: given a fixed centroid, `generate_explanation_label` always produces the same string.
    - **Low-sample behavior**: overall gate (n < 100 → run fails), per-cluster flag (event_count < 30 → `return_profile["low_sample"] == True`).
    - **Scanner filtering**: only events matching the requested `scanner_type` are clustered; the `SignalAnalysisRun.scanner_type` field is set correctly.

## Architecture / Approach

### §1 Schema changes

Two Alembic migrations (can be one revision):

```python
# In signal_analysis_runs
run_type = Column(String(50), nullable=True, index=True)  # "indicator" | "explanation_archetype"

# In scanner_events
archetype_cluster_id = Column(
    Integer, ForeignKey("signal_clusters.id"), nullable=True, index=True
)
```

Backfill `signal_analysis_runs.run_type = 'indicator'` for existing completed rows (the existing task will start writing `"indicator"` going forward, but historical rows remain recognizable).

### §2 StatisticalDiscoveryService — new method

Add `generate_explanation_label()` to `backend/app/services/statistical_discovery.py`:

```python
def generate_explanation_label(
    self,
    centroid: dict[str, float],
    global_mean: dict[str, float],
) -> str:
    """
    Like generate_label, but translates explanation feature names to plain English.
    Picks the top 2 features by absolute centroid deviation from the global mean.
    """
```

Feature name translation rules (applied before the qualifier):
- `criterion.{id}.passed` (boolean-as-float): centroid ≥ 0.5 → `"{id} criterion passed"`, else `"{id} criterion failed"`
- `criterion.{id}.observed` (numeric): "high {id}" or "low {id}" based on qualifier
- `warning.{code}` (boolean-as-float): centroid ≥ 0.5 → `"{code} warning"`, else `"no {code} warning"`
- `confidence_score`: "high confidence" or "low confidence"
- Unknown names fall back to the existing `"high {name}"`/`"low {name}"` format

The existing `generate_label` remains unchanged (used by `analyze_signal_features`).

### §3 New Celery task: `generate_signal_archetypes`

Location: `backend/app/tasks/quality.py` (add after `analyze_signal_features`)

```python
@celery_app.task(bind=True, max_retries=1, name="app.tasks.generate_signal_archetypes")
def generate_signal_archetypes(
    self, scanner_type: str | None = None, k: int = 6
) -> None:
```

Execution flow:

1. Create `SignalAnalysisRun(status="running", run_type="explanation_archetype", scanner_type=scanner_type, celery_task_id=...)`. Commit.

2. Query events:
   ```python
   db.query(ScannerEvent.id, ScannerEvent.scanner_type, ScannerEvent.explanation,
            ScannerOutcomeSnapshot.interval_key, ScannerOutcomeSnapshot.pct_change)
     .join(ScannerOutcomeSummary, ScannerOutcomeSummary.scanner_event_id == ScannerEvent.id)
     .join(ScannerOutcomeSnapshot, ScannerOutcomeSnapshot.scanner_event_id == ScannerEvent.id)
     .filter(
         ScannerEvent.explanation.isnot(None),
         ScannerOutcomeSummary.is_complete.is_(True),
         ScannerOutcomeSnapshot.status == "captured",
     )
   ```
   Apply `scanner_type` filter if provided.

3. Gate: if unique event count < 100, mark run failed and return.

4. Build feature DataFrame by calling `extract_explanation_features(rows)` from `app.services.explanation_features` (interface provided by #463). This returns a DataFrame with `event_id`, `interval_key`, `pct_change`, plus explanation-derived feature columns.

5. Call `svc.build_feature_matrix(df)` → drops sparse/non-numeric columns, same as existing task.

6. Call `svc.run_kmeans(df, k=k)` → `(cluster_labels, centroids)`.

7. Call `svc.compute_conditional_stats(df, cluster_labels)` → per-cluster return profiles.

8. Compute `global_mean` from feature columns.

9. For each cluster index `ci`:
   - `label = svc.generate_explanation_label(centroids[ci], global_mean)`
   - `event_count = sum(1 for v in cluster_labels.values() if v == ci)`
   - `return_profile = conditional_stats.get(ci, {})`
   - If `event_count < 30`: set `return_profile["low_sample"] = True`
   - Persist `SignalCluster(analysis_run_id=run.id, cluster_index=ci, label=label, centroid=centroids[ci], return_profile=return_profile, event_count=event_count)`

10. For each `(event_id, cluster_index)` in `cluster_labels`:
    ```python
    db.query(ScannerEvent).filter(ScannerEvent.id == event_id).update(
        {"archetype_cluster_id": cluster_id_map[cluster_index]},
        synchronize_session=False,
    )
    ```

11. Mark run `status="completed"`, `event_count=...`, `completed_at=utc_now()`. Commit.

Error handling: same try/except/finally pattern as `analyze_signal_features`.

`LatestAnalysisResponse` is reused as-is. The `feature_weights` field will be empty for archetype runs (no SHAP pass). `ClusterSummary.centroid` contains explanation feature names. The `low_sample` flag travels inside each cluster's `return_profile` dict (accessible as `cluster.return_profile.get("low_sample", False)` in Python; the existing `ClusterReturnInterval` schema does not need to change since `return_profile` is `dict[str, ClusterReturnInterval]` and `low_sample` is a top-level key alongside the interval keys).

**Note:** `ClusterSummary.return_profile` is declared as `dict[str, ClusterReturnInterval]` in `schemas/analysis.py`. The `low_sample` flag is a `bool`, not a `ClusterReturnInterval`. Options:
- Change `return_profile` to `dict[str, Any]` in the response schema (breaks type safety)
- Move `low_sample` to a new top-level field `ClusterSummary.low_sample: bool = False`

**Recommendation**: add `low_sample: bool = False` as a top-level field on `ClusterSummary`. It is populated from `cluster.return_profile.get("low_sample", False)` at serialization time and the JSONB storage is not changed.

## Alternatives Considered

### A. Extend `analyze_signal_features` instead of a new task

Add a second K-means pass using explanation features within the existing task. Rejected because:
- Explanation data is sparse while Epic 1 is still in progress; the two data populations have different maturity levels and should gate independently.
- `signal_cluster_id` would be overwritten on a single FK, causing race conditions if either task retriggers.
- Regression risk: the existing task is stable and produces correlations/SHAP used by EdgeExplorer. Entangling it with sparse explanation data could break that flow.

### B. Reuse `signal_cluster_id` instead of adding `archetype_cluster_id`

Have the archetype task write to the existing `signal_cluster_id` column. Rejected because:
- `analyze_signal_features` would overwrite archetype assignments on its next run (both tasks update the same column for overlapping events).
- Distinguishing which cluster type a `signal_cluster_id` points to requires an extra join to `SignalAnalysisRun.run_type`, adding query complexity everywhere `signal_cluster_id` is read.
- A dedicated `archetype_cluster_id` FK makes the data model self-documenting and avoids all sequencing concerns.

### C. Include outcome metrics (mfe_pct, mae_pct) in the K-means feature matrix

Treat `ScannerOutcomeSummary` columns as clustering inputs. Rejected as target leakage: the cluster's outcome profile is what we want to *discover* about each archetype, not what defines it. Outcome metrics are attached as the `return_profile` (same pattern as the existing task), not as clustering inputs.

## Open Questions (non-blocking)

1. **Interface for `extract_explanation_features()`** — the exact function signature and column naming convention from #463 are not yet finalized. The implementation of issue #466 should treat this interface as a hard dependency; if #463 produces a different DataFrame shape, the archetype task's input construction needs to adapt.

2. **Explanation-feature naming convention** — the archetype label translator assumes feature names like `criterion.{id}.passed`, `criterion.{id}.observed`, `warning.{code}`. If #463 uses a different prefix scheme (e.g., `expl_crit_{id}_passed`), the `generate_explanation_label` translation dict must be updated accordingly.

3. **k selection** — the default `k=6` matches the existing task. With explanation features that are more categorical (many boolean flags), the optimal k may differ. This can be tuned via the API `k` param and is out of scope for this issue.

4. **API exposure of `archetype_cluster_id`** — the existing `/api/v1/scanner/results` endpoint serializes `ScannerEvent` fields. Adding `archetype_cluster_id` to the response schema and the frontend event card is a follow-up (Epic #449 issue 6/9). This issue only needs to persist and expose the FK.

## Assumptions

- **[ASSUMPTION]** `scanner_events.explanation` JSONB column exists by the time #466 is implemented (added by Epic 1 issues, which are prerequisites of blocker #463).
- **[ASSUMPTION]** `extract_explanation_features()` from #463 returns a DataFrame where boolean fields are `float` (0.0/1.0), not Python `bool`, consistent with `StatisticalDiscoveryService.build_feature_matrix()` expectations.
- **[ASSUMPTION]** `ScannerOutcomeSummary.is_complete` is the correct completeness gate (reused from the existing task pattern).
- **[ASSUMPTION]** The `low_sample: true` flag lives at the top level of the `return_profile` JSONB dict, alongside interval keys like `"1d"`, `"5d"`. No schema migration is needed for this because `return_profile` is already a free-form `JSONB` column.
