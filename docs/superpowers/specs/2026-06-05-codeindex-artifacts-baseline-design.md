> Tracking issue: [#200](https://github.com/omniscient/markethawk/issues/200)

## Goal

`CLAUDE.md` and `DEVELOPMENT.md` describe `symbolindex.json` and `codeindex.json` as
"maintained in-repo (committed artifacts)" and instruct agents to use `lookup_symbol` and
`get_impact` as mandatory pre-steps. Neither file exists in the repo; `docs/codeindex-hotspots.md`
is a seed stub. Every agent that follows the instructions literally fails silently (or wastes a
tool call on a non-existent MCP response).

This spec resolves both halves of the discrepancy: commit a real baseline so the artifacts
actually exist, and soften the agent instructions to advisory-with-grep-fallback so they are
honest about when the tools are unavailable.

## Scope

**In scope:**
- Generate `codeindex.json`, `symbolindex.json`, and a real `docs/codeindex-hotspots.md` by
  running the codeindex CLI (already installed at `/usr/local/bin/codeindex` in the factory).
- Commit all three files.
- Update the four affected lines in `CLAUDE.md` (lines 116-119) to advisory wording.
- Update the three affected lines in `DEVELOPMENT.md` (lines 387-389) to advisory wording.

**Out of scope:**
- Changing the factory workflow nodes (`update-codeindex`, `regen-codeindex`) â€” they are already
  correct and non-fatal. This PR gives them a baseline to update rather than create from scratch.
- Adding or removing MCP server configuration â€” `.claude/settings.local.json` is already written
  by the factory entrypoint.
- Gitignore changes â€” the artifacts are not gitignored (confirmed).
- Adding codeindex to pre-commit for non-factory runs â€” the existing warn-only pre-commit hook
  (`codeindex high-blast`) is sufficient; adding a blocking hook would break dev machines without
  the tool.

## Architecture

### Step 1 â€” Generate artifacts

```bash
codeindex analyze .
codeindex symbols . --inline
codeindex high-blast > docs/codeindex-hotspots.md
```

`codeindex analyze .` writes `codeindex.json`; `codeindex symbols . --inline` writes
`symbolindex.json`. Both commands exit non-zero on failure and produce no partial output, so
a failed run leaves the repo clean.

### Step 2 â€” Commit

```bash
git add codeindex.json symbolindex.json docs/codeindex-hotspots.md
git commit -m "chore: commit initial codeindex baseline (#200)"
```

## Alternatives Considered

**Documentation-only (no artifact commit)**
Update CLAUDE.md and DEVELOPMENT.md to say the files are generated at factory runtime, without
committing a baseline. Technically satisfies the second acceptance criterion ("no mandatory step
that cannot be performed") but leaves the repo permanently absent of the artifacts until the
next successful factory run â€” which is the exact failure mode the issue calls out. Rejected
because codeindex is available in the factory environment and the additional cost is trivial.

**Commit artifacts only (no doc changes)**
Commit the files without updating the wording. The `lookup_symbol`/`get_impact` instructions
remain mandatory. This is incomplete: once the artifacts are present the MCP tools work inside
the factory, but agents outside the factory (plain Claude Code sessions) still hit the
"unsatisfiable step" problem if the MCP server isn't running. Rejected.

## Assumptions

- `codeindex analyze .` and `codeindex symbols . --inline` are the correct invocations for
  generating `codeindex.json` and `symbolindex.json` respectively (confirmed from
  `scripts/codeindex.sh` and the `update-codeindex` workflow node).
- `codeindex high-blast` produces usable hotspot output on the current codebase size.
- The committed artifacts are not sensitive (they index symbol names and call-graph edges â€”
  no secrets or data).

## Open Questions

None blocking. The factory's `regen-codeindex` node will overwrite these files on each future
implementation run. If the hotspots list turns out to be empty (no high-blast files detected),
`docs/codeindex-hotspots.md` will contain the tool's empty output, which is still better than
the current seed stub.
