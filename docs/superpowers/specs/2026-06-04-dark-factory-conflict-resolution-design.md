# Dark Factory — Auto-Detect and Resolve Merge Conflicts

> Tracking issue: [#210](https://github.com/omniscient/markethawk/issues/210)

## Goal

Add a dark-factory enhancement that **automatically detects and resolves merge conflicts**
on in-review PRs, so feature branches that drift behind `main` get reconciled without a
human babysitting the merge button.

Today, merge conflicts are only discovered at `Close issue #N` time — `gh pr merge` fails
and the issue lands in **Blocked** with no attempt to fix it. Feature branches never sync
with `main`, so a PR sitting in review silently rots into conflict as other PRs land.

## Scope

**In scope:**

- **Priority 1.5 scheduler gate** (`scheduler.sh`): new loop that scans in-review PRs for
  `mergeable == CONFLICTING` and dispatches a conflict-resolution run each cycle.
- **New `resolve` workflow intent** (`archon-dark-factory.yaml`): checkout branch → de-conflict
  → validate → push, with no implement step.
- **`de-conflict` bash node** (shared between `resolve` and `continue` flows): merge `main`
  into branch, run Tier 1 mechanical resolution, optionally run Tier 2 AI resolution, enforce
  hard grep gate for surviving conflict markers, write artifact.
- **Config kill-switch** (`.claude/skills/refinement/config.yaml`): `conflict_resolution.enabled`
  and `conflict_resolution.ai_tier` flags.
- **Tier 1** (mechanical, no AI): regenerate-from-source artifacts (`codeindex.json`,
  `symbolindex.json`, `docs/codeindex-hotspots.md`), regenerate-from-manifest
  (`package-lock.json`), tool-merge for migration heads (`alembic merge heads`), and
  guarded union-merge for `backend/app/models/__init__.py`. Identified by an explicit
  **path allowlist** — Tier 1 structurally cannot touch real source logic.
- **Tier 2** (AI): only files still carrying conflict markers after Tier 1, resolved by
  Claude with issue + PR + `git log` context, preserving both intents.
- **Tier 3** (escalate): any surviving conflict marker (hard grep gate), or a failed
  post-merge validate → Blocked + comment.
- **Artifact**: `$ARTIFACTS_DIR/conflict_resolution.md` surfaced in the issue report.
- **Scheduler tests**: extend `test_scheduler.sh` to cover the Priority 1.5 gate.

**Out of scope:**

- Rebasing or force-pushing: merge only (`git merge main`, never `git rebase`).
- Resolving conflicts on Blocked items (Priority 1.5 targets In Review only — the existing
  Priority 3 Blocked retry handles those).
- Changes to the refine/plan flows.

## Architecture

### Trigger: Priority 1.5 Scheduler Gate

A new loop in `scheduler.sh`, inserted between Priority 1 (In Review comments) and
Priority 2 (Ready items). Each cycle:

1. Reads the `conflict_resolution.enabled` flag from `.claude/skills/refinement/config.yaml`
   via `yq` or `python3 -c`.
2. For each In Review item (that passes `has_skip_label` check):
   a. Fetch `mergeable` for the PR via `gh pr view $PR_NUM --json mergeable --jq '.mergeable'`.
   b. `CONFLICTING` → dispatch `"Resolve conflicts issue #N"`.
   c. `UNKNOWN` → skip (GitHub hasn't computed it yet).
   d. `MERGEABLE` → skip.
3. Uses the existing `dispatch()`, `increment_retry`, `trip_to_blocked` pattern, with a
   dedicated `${issue_num}:resolve` retry key.
4. Emits a structured `conflict_gate` log line per in-review item scanned.
5. Breaks after first dispatch (one factory run at a time).

The Priority 1.5 loop does NOT re-run for items already dispatched this cycle (tracked
via the `DISPATCHED` variable).

### New `resolve` Intent in Workflow

`parse-intent` enum extended to include `resolve`. The workflow node graph for `resolve`:

```
parse-intent
  ↓
fetch-issue
  ↓
setup-branch-resolve  (checkout existing feat/issue-N-slug branch)
  ↓
de-conflict           (merge main → Tier 1 → Tier 2 → hard grep gate)
  ↓
preview-up            (use existing preview stack if up; else skip preview)
  ↓
validate              (existing validate command, PREVIEW_SKIPPED=true if no stack)
  ↓
push-resolve          (git push to existing PR — no PR creation)
  ↓
status-stays-in-review (no board move — issue stays In Review)
  ↓
report                (includes conflict_resolution.md summary)
```

For the `continue` flow, `de-conflict` is inserted **after** `regen-codeindex` and
**before** `preview-up`, so iterations de-conflict in the same pass they already validate + push:

```
regen-codeindex → de-conflict → preview-changeset → classify-preview → preview-up → validate
```

### De-conflict Node (Bash)

```
BEFORE: git merge main (--no-edit, no rebase, no force-push)
IF NO CONFLICTS: done (writes conflict_resolution.md with verdict=none)

TIER 1 (mechanical):
  For each conflicted file in the allowlist:
    - codeindex.json, symbolindex.json, docs/codeindex-hotspots.md:
        git checkout --theirs <file>
        (these are regenerated by regen-codeindex after every implement run — theirs is correct)
    - package-lock.json:
        git checkout --theirs package-lock.json
        cd frontend && npm install --package-lock-only
        git add package-lock.json
    - alembic/versions/*.py (migration head divergence):
        python -m alembic merge heads -m "merge conflict resolution"
        git add alembic/versions/
    - backend/app/models/__init__.py:
        union-merge: extract all `from .X import Y` lines from both sides (HEAD and MERGE_HEAD),
        deduplicate, sort, write back; git add
  git commit -m "chore(#N): Tier-1 conflict resolution (merge main)"

TIER 2 (AI — only if conflict_resolution.ai_tier: true AND files still have markers):
  For each remaining conflict file:
    - Provide Claude with: issue body, PR description, git log --oneline main..HEAD,
      the conflicted file diff, and the context of both sides.
    - Claude resolves each file, preserving both intents.
    - git add <resolved file>
  git commit -m "chore(#N): Tier-2 AI conflict resolution"

HARD GREP GATE:
  REMAINING=$(grep -rn "<<<<<<< " . --include="*.py" --include="*.ts" --include="*.tsx" \
    --include="*.json" --include="*.yaml" --include="*.yml" --include="*.sh" \
    --include="*.sql" --include="*.md" 2>/dev/null || true)
  if [ -n "$REMAINING" ]; then
    echo "CONFLICT_VERDICT=escalate" >> $ARTIFACTS_DIR/conflict_resolution.md
    # trip_to_blocked handled by the validate step failing
    exit 1
  fi

ARTIFACT: conflict_resolution.md
  FILES_CONFLICTED: N
  TIER1_RESOLVED: N
  TIER2_RESOLVED: N
  ESCALATED: 0|1
  VERDICT: none | tier1 | tier2 | escalate
```

### `push-resolve` Node

For the `resolve` intent only — no PR creation (the PR already exists). Just `git push`.
On non-zero exit: calls the equivalent of `trip_to_blocked` for the issue and exits.

### `status-stays-in-review` vs `status-in-review`

After a successful resolve run, the issue **stays In Review** — no board move. The existing
`status-in-review` node is re-used for `resolve` but is effectively a no-op (it already
moves to In Review, which is already the current status).

### Report Integration

The `report` node in the workflow reads `$ARTIFACTS_DIR/conflict_resolution.md` and
appends a `### Conflict Resolution` section when it exists:

```
### Conflict Resolution

- Files in conflict: 3
- Tier 1 resolved: 2 (codeindex.json, symbolindex.json)
- Tier 2 resolved: 1 (backend/app/services/scanner.py)
- Verdict: tier2
```

### Kill-Switch and Config

In `.claude/skills/refinement/config.yaml`:

```yaml
conflict_resolution:
  enabled: true        # false = Priority 1.5 loop is a no-op
  ai_tier: true        # false = skip Tier 2 entirely (Tier 1 + escalate only)
```

The scheduler reads `enabled` via `python3 -c "import yaml; ..."`. If `enabled` is `false`
or the config key is absent, the Priority 1.5 loop is skipped entirely.

### Retry / Circuit-Breaker

Uses the existing `get_retry_count / increment_retry / trip_to_blocked` machinery:
- Retry key: `${issue_num}:resolve`
- Max retries: `$MAX_RETRIES` (same as other phases)
- On trip: moves to Blocked, adds `needs-discussion`, posts explanatory comment.

### Guardrails

- `UNKNOWN` mergeable is always skipped — GitHub may still be computing it.
- `SKIP_LABELS` (needs-discussion, epic) are honored in Priority 1.5.
- Priority 1.5 breaks after first dispatch (factory concurrency guard is already checked
  before entering Priority 1.5).
- No force-push: always `git push` (fast-forward after merge).
- After a failed validate, nothing is pushed — the issue goes to Blocked via the same
  on_failure path as any other failed factory run.

## Requirements

| ID | Requirement |
|----|-------------|
| R1 | Each scheduler cycle, in-review PRs with `mergeable == CONFLICTING` are identified and dispatched for resolution. |
| R2 | `UNKNOWN` mergeable status is skipped; `MERGEABLE` is a no-op. |
| R3 | Tier 1 resolves generated/tool-merge files mechanically, using a path allowlist that cannot touch application logic. |
| R4 | Tier 2 uses Claude to resolve remaining conflict markers only when `conflict_resolution.ai_tier: true`. |
| R5 | A hard grep gate (`<<<<<<< `) runs after all tiers; any surviving marker escalates to Blocked. |
| R6 | After resolution, the existing `validate` gate (pytest + tsc + endpoint curl) runs. Failure escalates to Blocked. |
| R7 | `conflict_resolution.enabled: false` disables the Priority 1.5 loop as a kill-switch. |
| R8 | After a successful resolve run, the issue stays In Review (no board move). |
| R9 | The `de-conflict` node is shared: `continue` runs it before validate, preventing silent drift. |
| R10 | Retry / circuit-breaker for `:resolve` key uses existing machinery with `MAX_RETRIES` limit. |
| R11 | A `conflict_gate` structured log line is emitted per in-review item scanned. |
| R12 | `$ARTIFACTS_DIR/conflict_resolution.md` is written by the de-conflict node and included in the issue report. |

## Alternatives Considered

### Rebase instead of merge
Rebasing rewrites commit history, requires force-push, and is disruptive for PRs under
review. Merge is the only safe option.

### Handle conflicts only at Close time
This is the current (broken) behavior. It fails silently while the branch rots for days.
Proactive detection in Priority 1.5 catches drift while it is small and fresh.

### AI-only resolution (no Tier 1)
Generated artifacts (`codeindex.json`, `package-lock.json`) should be regenerated, not
AI-resolved. Regeneration is deterministic and does not hallucinate. Tier 1 keeps AI costs
low and avoids non-deterministic output in binary/generated files.

### Add a new top-level Dark Factory command (not a shared de-conflict node)
Duplicating resolution logic into a standalone command would create drift between the
standalone and the continue-integrated paths. A shared de-conflict node in the workflow
is the single source of truth.

## Assumptions

- `gh pr view $PR_NUM --json mergeable` returns `CONFLICTING`, `MERGEABLE`, or `UNKNOWN`.
- The factory container has `python3`, `jq`, `yq` or `python3 -c "import yaml"` available.
- A `resolve` run for a PR that has since been closed or merged is a no-op (the dispatch
  will fail gracefully when the branch or PR is not found).
- `alembic merge heads` works when there are exactly two migration heads; if there are
  more than two, the command should be called once and handles all heads.
