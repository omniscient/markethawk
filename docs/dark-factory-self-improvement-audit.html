Dark Factory — Self-Improvement Loop Audit

Issue: #213  |  Epic: #262 — Harden the self-improvement loop  |  Date: 2026-06-09  |  Reference: Addy Osmani — Self-Improving Agents

This report audits the MarketHawk Dark Factory pipeline against the self-improving agent patterns described in Addy Osmani's article. Each pattern is mapped to the current Dark Factory mechanism, gaps are ranked by impact, and the two phases of this issue's remediation are described in full.

2. Gap Analysis

Gaps ordered by impact. Only ❌ Missing and ⚠️ Partial rows appear here.

Iterative Feedback: Gate Failures Not Written to Memory

Gap: The conformance and code-review gates block the pipeline when they find violations, but those lessons are never written back to .archon/memory/. Future runs can repeat the same class of mistake because no [AVOID] entry was created to warn them.

Failure mode without it: The same structural mistake (e.g., "don't append to memory files with raw >>") recurs in a later issue's implementation. The conformance gate catches it again, the reconcile loop fixes it again — the same fix is applied twice without ever teaching the system. Each gate-catch is a one-off correction, not a compound learning event.

Disposition: Addressed by this issue, Phase 2. Conformance MATERIAL verdict → [AVOID] entry via write_memory_entry. Code-review BLOCKED status → [AVOID] entry with source:code-review attribution. Ships when Phase 2 changes are merged.

Memory Effectiveness Tracking: [AVOID] Recurrence Not Detected

Gap: There is no mechanism to detect whether a pattern marked [AVOID] has been triggered (violated) in a subsequent run. The pipeline writes the lesson but never confirms it was learned.

Failure mode without it: The memory system accumulates [AVOID] entries that may or may not be working. A persistent recurrence of the same mistake (e.g., across multiple issues) would go unnoticed until a human reviews the commit history manually. The feedback loop is open — there is no signal confirming closure.

Disposition: No issue filed yet. A detection mechanism could grep each new issue's diff against known [AVOID] entries and flag matches at Phase 2 PLAN time. Medium-term work item.

Scoped Memory Loading: Entry-Level Path-Tag Filtering Missing

Gap: Memory files are loaded by area (backend / frontend / infra) but not filtered by entry. Once gate stages start auto-writing [AVOID] entries (Phase 2 of this issue), a backend-specific entry will load even when a frontend-only change is being reviewed — adding noise and diluting signal.

Failure mode without it: As the pipeline writes more [AVOID] entries automatically (from conformance/code-review gates), the memory files grow and every entry loads regardless of whether it's relevant to the current issue. This recreates the "context bloat" problem that #254 addressed at the file level, but now at the entry level.

Disposition: Addressed by this issue, Phase 2. The load_memory() function filters path-tagged entries against the issue's affected file set. Untagged entries always load (backward-compatible). Ships when Phase 2 changes are merged.

Parallel Multi-Agent Orchestration: Sequential Within Single Issue

Gap: The pipeline is sequential within a single issue — refine → plan → implement → conformance → code-review. Independent substages (e.g., backend unit tests and frontend type-checks) run sequentially rather than in parallel.

Failure mode without it: Large issues with both backend and frontend changes take longer than necessary. For this project's issue sizes (S/M/L), the impact is low — most issues touch one area. Parallelism exists across issues (Archon worktrees) but not within a single pipeline run.

Disposition: Low priority given bounded issue scope. No issue filed. The article itself notes that deepening a single agent is often more effective than managing complex swarms.

Periodic Fresh Planning Restarts: No Automatic Mid-Run Reset

Gap: There is no automatic mechanism to re-run the planning phase mid-implementation when the scope or approach diverges from the original plan.

Failure mode without it: For large issues, the implement agent may drift from the plan in later tasks. The conformance gate catches the drift post-implementation, but an earlier correction would save a full conformance cycle. Impact is minimal for S/M issues.

Disposition: Not prioritized. The "continue" flow (feedback on PR → re-run implement) serves as a manual reset trigger. The conformance reconcile loop (up to 3 cycles) handles implementation drift. No issue filed.

Dev-Browser Skill: No Automated UI Verification

Gap: Frontend changes are validated by TypeScript type-checking only. There is no headless browser skill to verify that UI flows work end-to-end after a change.

Failure mode without it: A frontend change that passes tsc --noEmit but breaks a React Query hook or a component interaction would go undetected until a human tests the UI.

Disposition: No issue filed. MarketHawk's core value is backend scanner logic, so this gap has limited blast radius. A Playwright-based verify command would be the natural next step.

3. What This Issue Addresses

Phase 1 (ships with this PR):

  • This HTML audit report — maps all 19 article patterns against the Dark Factory with status and gap analysis

Phase 2 (gated on #254 — now Done, ships with this PR):

  • Conformance gate → [AVOID] memory write on MATERIAL verdict (resolved violations become lessons)
  • Code-review gate → [AVOID] memory write on BLOCKED status (blocking findings become lessons)
  • path: tag filtering in implement/refine/plan load phases (entry-level memory scoping)
  • Unit tests: dark-factory/tests/test_conformance_memory_write.sh, test_load_memory.sh
Note on write_memory_entry: The plan anticipated that #254 would deliver a write_memory_entry shell function. #254 instead implemented the write rules inline in the implement command's Phase 5. This issue defines write_memory_entry as an inline shell function in each gate command file, implementing #254's write contract (dedup, cap, TTL format) directly. No external dependency required.

4. Deferred / Non-Goals

  • Memory effectiveness tracking (detecting [AVOID] recurrence) — no issue yet; medium-term
  • Pipeline metrics dashboards — issue #212 covers this
  • Retroactively path-tagging existing memory entries — deferred; entries without path: always load
  • Auto-fixing conformance violations via additional reconcile loops — deferred
  • Vector database / semantic memory retrieval — intentionally out of scope; flat-file reading is preferred at this scale
  • Dev-browser skill (headless UI verification) — not prioritized
  • Automatic mid-run planning restarts — not prioritized; conformance reconcile loop handles drift
