# Canonical Module Map Design Spec

**Issue**: #169 — docs: one canonical module map (de-duplicate ARCHITECTURE / PROJECT_STRUCTURE / database-schema)
**Date**: 2026-06-04
**Component**: `ARCHITECTURE.md` · `PROJECT_STRUCTURE.md`

## Summary

Strip all inline prose annotations from `PROJECT_STRUCTURE.md`'s directory tree, making it a paths-only structure. `ARCHITECTURE.md` is the single canonical source for per-file responsibility descriptions. `Docs/database-schema.md` already auto-generates and requires no change. `symbolindex.json` is machine-maintained and unchanged.

## Requirements

- `PROJECT_STRUCTURE.md` tree must contain no inline `# prose comments` on file or directory lines after this change
- Directory and file names in the tree remain unchanged; only the trailing `# ...` suffix is removed
- The "Notes for Navigation" section at the bottom of `PROJECT_STRUCTURE.md` is preserved
- `ARCHITECTURE.md` is unchanged — it already owns the Backend Module Map tables as the canonical source
- `Docs/database-schema.md` is unchanged — it is auto-generated by `backend/scripts/generate_schema_doc.py`
- `symbolindex.json` is unchanged — machine-maintained by the codeindex tool
- No new scripts, generators, or tooling are introduced

## Architecture

**Single source of truth after this change:**

| What you want to know | Go here |
|-----------------------|---------|
| Per-file responsibilities (prose) | `ARCHITECTURE.md` — Backend Module Map tables |
| Where files live (paths) | `PROJECT_STRUCTURE.md` — paths-only tree |
| Database schema (ERD, columns, FK) | `Docs/database-schema.md` — auto-generated Mermaid |

**Why paths-only is sufficient:** Developers navigating the tree need to locate files, not read descriptions — the IDE and `ARCHITECTURE.md` answer "what does this do". Removing prose from the tree eliminates the drift surface: three copies of the same description written once.

**Scope boundary:** Generating `PROJECT_STRUCTURE.md` from `symbolindex.json` would produce the same paths-only result but adds a generation script to maintain. That is out of scope. Manual paths-only editing is equivalent and operationally simpler.

## Alternatives Considered

- **Generate PROJECT_STRUCTURE.md from symbolindex.json**: Equivalent result, higher maintenance (generation script, CI step). Out of scope for this issue.
- **Keep prose in both docs with a sync check**: Still two write paths; automation can break or lag. Rejected — single source is cleaner.
