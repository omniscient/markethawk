# codeindex × Dark Factory Integration — Design Spec

**Status:** spec-pending-review

## Summary

Integrate the [codeindex](https://github.com/scheidydude/codeindex) dependency analyzer into the dark factory and the local developer workflow. codeindex builds a dependency graph with per-file **blast-radius** scores (`codeindex.json`) and a **symbol map** (`symbolindex.json`), and exposes them five ways: CLI, markdown report, MCP server, pre-commit hook, and CLAUDE.md injection.

Two audiences:

- **The autonomous agent** — codes with blast-radius awareness (`get_impact`) and O(1) symbol lookups (`lookup_symbol`) instead of grepping ~200 files, via a project-scoped **MCP server** plus a short CLAUDE.md hint (the README's "hybrid" workflow).
- **Human developers** — an interactive local **visualization** (`serve --viz`), a per-PR **blast-radius report**, and a tracked **high-blast hotspots** doc.

## Why #107 failed (carry-forward context)

The #107 plan added codeindex to `backend/requirements.txt`. That file is `COPY`'d and `pip install`'d into the `python:3.12-slim` backend image (`backend/Dockerfile:13-14`) — an image with **no `git` binary**. `pip install git+https://github.com/scheidydude/codeindex.git` therefore failed during every `preview-up` image rebuild, the backend never came up, and the run was Blocked four times.

Two traps this spec avoids:

1. **`pip install codeindex` installs the wrong package.** The PyPI name `codeindex` is an unrelated project (`cr0hn/the-opensource-context`, a "semantic search client"). The intended tool is only installable from git: `pip install "git+https://github.com/scheidydude/codeindex.git"`.
2. **codeindex is agent/dev tooling, not an app runtime dependency.** It belongs in the dark-factory image (`ubuntu:24.04`, which already installs `git`), never in `backend/requirements.txt`.

## Design decisions (agreed in brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Agent consumption | **Hybrid: MCP server + CLAUDE.md hint** | README's pick for active coding; tools on demand, near-zero per-prompt token cost. |
| Index lifecycle | **Committed artifact** | Present on every checkout (instant viz), diffs show structural drift. Regenerated by the factory; see freshness model below. |
| `impact` enforcement | **Advisory + warn-only hook** | Guidance via MCP/CLAUDE.md; a non-blocking pre-commit warning for visibility. A strict gate could turn a legitimate high-blast change into a failed autonomous run. |
| Human outputs | **Local viz + per-PR blast report + hotspots doc** | Viz is inherently interactive/local; reports/hotspots give reviewers and the team ongoing insight. No persistent hosted viz this iteration. |

## Components

### A. Install — `dark-factory/Dockerfile`

Add, after the existing system-deps layer (git already present):

```dockerfile
RUN pip install "git+https://github.com/scheidydude/codeindex.git"
```

No extras (`[watch]`, `[yaml]`, `[toml]`) — the tool is pure-stdlib and the factory does not need `--watch`. Resolve the install path with `which codeindex` for the MCP config (see B).

### B. Agent consumption — Hybrid (MCP + CLAUDE.md hint)

**MCP wiring.** After cloning, `entrypoint.sh` writes the **cloned repo's** `.claude/settings.local.json` (gitignored — see `.gitignore:41`) registering the MCP server with codeindex's **absolute path**:

```jsonc
{
  "mcpServers": {
    "codeindex": { "command": "<abs-path>/codeindex", "args": ["serve", "--mcp"] }
  }
}
```

The README explicitly warns a bare `"codeindex"` command fails because Claude Code does not inherit the shell PATH — the absolute path (from `which codeindex`) is required. Using the **local, gitignored** settings file means the registration is available to the implement-step agent but is **never committed** and is **not imposed on human developers** who may not have codeindex installed.

This gives the agent all 6 MCP tools: `analyze_repo`, `build_symbol_index`, `lookup_symbol`, `get_impact`, `get_dependencies`, `get_high_blast_files`.

**CLAUDE.md hint.** A committed, ~3-line `## Codeindex` section (the README's hybrid hint):

```markdown
## Codeindex
Symbol index `symbolindex.json` + dependency index `codeindex.json` are maintained in-repo.
Use the `lookup_symbol` MCP tool before grepping for any function/class.
Use `get_impact` before modifying a high-blast file.
```

Costs almost no tokens; primes the agent to reach for the tools rather than defaulting to grep.

**Implement command.** Add one instruction to `.archon/commands/dark-factory-implement.md` (Phase 3) to consult `get_impact` before editing high-blast files.

### C. Index lifecycle — committed artifact, fresh at two points

The repo is cloned fresh on every run, so the index must be regenerated during the run regardless. Two regeneration points:

1. **Startup (before `implement`)** — a new `update-codeindex` workflow node, dependent on `setup-branch`, runs:
   ```bash
   codeindex analyze . && codeindex symbols . --inline
   ```
   so the MCP tools serve a fresh index *during* coding. Scope: repo root with standard exclusions (`node_modules`, `.venv`, `dist`, `.git`, `__pycache__`, `*.egg-info`).
2. **Post-`implement` (before `push-and-pr`)** — regenerate **and commit** `codeindex.json` / `symbolindex.json` so the committed artifact matches the PR's final code (otherwise it is stale the moment the agent edits anything).

Remove `codeindex.json` / `symbolindex.json` from `.gitignore` (they are tracked now). Seed once on `main` (the first factory run commits them, or seed manually).

### D. `impact` enforcement — advisory + warn-only hook

Advisory behaviour is delivered by B. For visibility, add a **`repo: local`** hook entry to `.pre-commit-config.yaml` (alongside the existing `ruff` and `eslint` hooks) that runs a codeindex blast-score check on staged files and **always exits 0** (warn-only):

```yaml
  - repo: local
    hooks:
      - id: codeindex-blast
        name: codeindex blast-radius warning
        language: system
        entry: bash -c '<codeindex blast check on staged files> || true'
        pass_filenames: true
```

> **Wrinkle:** the factory does not currently run `pre-commit install`, so framework-managed hooks never fire there. `entrypoint.sh` must run `pre-commit install` in the clone for the warning to appear in the run log. Human devs already get it via the documented `pre-commit install` setup step. We use the `pre-commit` framework (not codeindex's own `install-hook`, which writes a raw `.git/hooks/pre-commit` that would clobber the framework-managed hook).

### E. Human outputs

- **Local viz** — `scripts/codeindex.sh`: generates the index and launches `codeindex serve --viz` (2D/3D graph, dependency matrix, treemap) on `:8080`. Documented in `CLAUDE.md` with the local `pip install "git+…"` one-liner. `scripts/` does not yet exist; the script creates it.
- **Per-PR blast-radius report** — the `push-and-pr` node runs `codeindex impact` / `high-blast` over the files in `git diff main...HEAD --name-only` and appends a **"Blast radius"** section to the PR body for reviewers.
- **Hotspots doc** — the post-implement node writes `codeindex high-blast --threshold N` output into a tracked `docs/codeindex-hotspots.md` so architectural risk hotspots stay visible over time.

## Testing

Presence/smoke checks appropriate to infra/config work (matches the architect's note on #107 — no app logic under test):

- `codeindex --version` succeeds in the built dark-factory image.
- `codeindex analyze` / `symbols` produce `codeindex.json` / `symbolindex.json`.
- MCP server registered in the clone's `.claude/settings.local.json` (absolute path present).
- `CLAUDE.md` contains the `## Codeindex` section.
- `.pre-commit-config.yaml` contains the `codeindex-blast` hook; it exits 0 on a high-blast staged file.
- A factory run produces a PR whose body contains a **Blast radius** section.
- `backend/requirements.txt` is unchanged; the preview build still succeeds.

## Must verify during planning (these sank #107 — do not assume)

1. Exact CLI subcommands/flags via `codeindex --help` **in the built image** — `analyze`, `symbols --inline`, `impact [--out|--json]`, `high-blast --threshold`, `serve --mcp|--viz`. The #107 architect blocked on unverified flags.
2. The warn-only pre-commit invocation against codeindex's `hook.py` / CLI (how to check staged files and force exit 0).
3. The absolute install path of `codeindex` in the `ubuntu:24.04` image (for the MCP `command`).
4. That `codeindex serve --mcp` runs headless and its 6 tools actually register in Claude Code (`claude mcp list` inside the container, or a smoke call).

## Out of scope

- Persistent hosted viz service.
- Any change to `backend/requirements.txt` or the backend / preview / prod runtime images.
- Pinning codeindex to a specific commit (advisory from #107; revisit if upstream churn bites).
